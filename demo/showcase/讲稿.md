# 演示讲稿 — 直接版本化 agent 的原始 session

> 每一步:**敲什么 · 屏幕出现什么 · 代表什么能力 · 可念的话**。屏幕输出取自真实运行。
>
> 核心一句话:**我们不设计什么 fact/schema —— Claude 自己把整个会话 dump 到磁盘,
> 我们就版本化那坨、push/pull 那坨;合并时让 agent 读懂对面的会话、自行合并,只有真冲突才问你。**

## 舞台

```sh
./demo/showcase/setup.sh
export PATH="/tmp/agit-demo/bin:$PATH"
```

建好:团队远端、**Alice 的仓库**(一条会话主张 `user_id`)、**Bob 的仓库**(一条会话主张 `uid`,和 Alice 冲突)。

---

# 第一幕 · Alice:把自己的 session 发布给团队

```sh
cd /tmp/agit-demo/showcase-alice
```

### ① 建 Agent Store —— context 和代码分开
```sh
agit init
```
```text
agit 已就绪。
  Environment : .../showcase-alice
  Agent Store : .../showcase-alice/.agit/agent
```
> 讲:在代码仓库旁建了第二个 git 仓库 `.agit/agent`,专门装 agent 的会话,`.agit/` 自动进 gitignore。**不侵入你的代码。**

### ② 镜像 Claude 的 session dump —— 零设计,直接 dump 全部
```sh
agit -a snap
```
```text
已镜像 claude-code 的 session dump:
  来源  : ~/.claude/projects/-...-showcase-alice
  写入  : .agit/agent/sessions/claude-code
  文件  : 1 个(0 更新 / 1 新增),... 字节
```
> 讲:Claude Code 本来就把整个会话(转录 jsonl + 子 agent + 工具结果)dump 到 `~/.claude/projects/<项目>/`。
> `sync` 就是把这坨镜像进来。**没有 fact、没有证据、没有 schema —— 就是原始会话。**
```sh
find .agit/agent/sessions -type f
```

### ③ 发布 —— Agent Store 就是 git 仓库
```sh
agit -a add -A && agit -a commit -m 'alice 的会话'
agit -a remote add origin http://team/showcase-origin.git   # demo 里用本地路径
agit -a push -u origin main
```
```text
扫描 1 个文件，未发现密钥。
 * [new branch]      main -> main
```
> 讲:push 走真的 git。**push 前自动扫了一遍密钥** —— 因为 dump 全部会话意味着 agent cat 过的
> `.env`、打印过的连接串都在转录里,不扫就发给全队了。(session 里的 UUID/requestId 那种高熵串不会误报,只报真密钥。)

# 第二幕 · Bob:一条命令拿到团队的会话

```sh
cd /tmp/agit-demo/showcase-bob
agit clone http://team/showcase-origin.git    # demo 里用本地路径
```
```text
已拉取 Agent Store ← .../showcase-origin.git
agit 已就绪。
```
> 讲:一条命令 —— clone 团队的 Agent Store 到 `.agit/agent` 并装好 hook。现在 Bob 手里既有 Alice 的会话,也能加自己的。

```sh
agit -a snap                              # 把 Bob 自己的会话也镜像进来
agit -a add -A && agit -a commit -m 'bob 的会话'
agit -a fetch origin
```
> 讲:Bob 现在本地有两边的会话文件(不同 uuid,天然不撞车)。

---

# 第三幕 · reconcile:让 agent 读对面的会话、自行合并,真冲突才问人

```sh
agit -a reconcile origin/main
```
```text
并入 1 条来自 origin/main 的 session。
让 claude 读两边会话、合成统一上下文……
统一上下文 → .../showcase-bob/CLAUDE.md

有 1 处需要你裁决的矛盾：
  1. 用户标识字段名不一致：alice 认定全库统一用 user_id，bob 决定改用 uid，
     二者对同一字段给出互斥的名字

改定后编辑 CLAUDE.md 的受管区块，或重跑 reconcile。
```
> 讲:这是全场核心。agit 把两边的会话喂给一个 agent(默认本机 claude),让它**读懂 Alice 那次会话
> 干了什么、什么约定**,和 Bob 的合成一份统一的工作上下文,写进 `CLAUDE.md`。
> **它自己能合的都合了,只把真正矛盾的点拎出来问你** —— 这里就是 user_id vs uid。

```sh
sed -n '/agit:begin/,/agit:end/p' CLAUDE.md
```
> 讲:生成的统一上下文有目标、已知结论、双方的约定/风格、下一步。下一个 Claude Code 会话自动加载它,
> **一开工就带着两个人合起来的上下文**。措辞不同但意思一致的会被合成一条,不会当成冲突。

# 这一场代表的能力

| 能力 | 在哪一步 |
|---|---|
| **最小侵入**:不设计 fact/schema,直接版本化 Claude 自己 dump 的会话 | ② |
| 两个库:context 与代码分开、互不污染 | ① |
| Agent Store 就是 git:push/pull/clone 全白拿 | ③②幕二 |
| **密钥防线**:dump 全会话也不把密钥发出去(且不被 UUID 噪声淹没) | ③ |
| **agent 驱动的语义合并**:读懂对面、自行合并 | 第三幕 |
| **真冲突才问人**:能合的都合,只拎矛盾 | 第三幕 |
| 装回 Claude Code:合成的上下文进 CLAUDE.md,下个会话自动带 | 第三幕 |
| 换 LLM 后端(Codex 口子):`AGIT_LLM_CMD=...` | — |
