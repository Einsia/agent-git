# 演示讲稿:版本化 agent 的原始 session,并通过对话合并两个 agent

> 每一步包含四部分:所输入的命令、屏幕输出、所体现的能力、可用的讲解话术。屏幕输出取自真实运行;对话由模型驱动,措辞每次略有差异。
>
> 核心结论:本工具不定义 fact 或 schema。运行时会将整个会话转储到磁盘,本工具即对该转储进行版本化,并以 push/pull 同步。**agent 是一段记忆**,以身份为键、独立于任何一个代码仓库存放;当两个 agent 对同一份代码产生分歧时,将两侧会话同时复活,令其各自阅读对方代码并自行合并,仅在出现真实冲突时才请求人工裁决。

## 准备

```sh
./demo/showcase/setup.sh
export PATH="/tmp/agit-demo/bin:$PATH"
export AGIT_HOME="/tmp/agit-demo/agit-home"   # 演示用 agent 放这里,不碰你真实的 ~/.agit
cd /tmp/agit-demo/showcase
# 按本讲稿逐幕演示
```

`setup.sh` 会构建一个仓库、两条分叉分支,以及**两个 agent**,各自对应一份工作:

- **feature-a**:`ratelimit` agent 新增登录限流器 `loginKey()`,以 `user_id` 字段作为分桶键。
- **feature-b**:`identity` agent 将身份字段 `user_id` 重命名为 `uid`。

两侧改动构成交叉冲突:`identity` 重命名的字段,正是 `ratelimit` 的新代码仍用于分桶的字段。

# 第一幕:agent 是一段记忆,由身份定位

### ① 两个 agent,两个 store
```sh
agit a list
```
```text
  identity   agt_493a5253-8c04-75e9-8a3b-39ce9e0b2266  —
● ratelimit  agt_3f17824b-0bba-74c2-804e-6ccb4c6f8bd0  —
```
> 讲解:agent 不是人,也不是项目里的一个文件夹,而是**一段记忆**——一个装满转录的 git 仓库,以**它知道什么**命名(`ratelimit` 知道限流,`identity` 知道身份字段)。`●` 是本 worktree 当前默认的 agent,这是默认值而非锁:两个 agent 可以在同一仓库中同时运行,用 `--agent` 逐次选择。

### ② 身份是 aid,store 以 aid 为键
```sh
agit a info ratelimit
```
```text
name   ratelimit
aid    agt_3f17824b-0bba-74c2-804e-6ccb4c6f8bd0
store  /tmp/agit-demo/agit-home/agents/agt_3f17824b-0bba-74c2-804e-6ccb4c6f8bd0
remote — (local only; agit a publish adds one)
```
> 讲解:身份是 `aid`,铸造一次,提交在 store 自己的 `agent.toml` 里。**不是名字**(名字是标签,会撞车),**也不是 URL**(URL 只是定位符——你在任何 remote 存在之前就已经铸造了这个 agent)。store 以 aid 为键存放在 `$AGIT_HOME/agents/<aid>/`,因此改名和发布都只是元数据改动,目录永不移动,正在运行的 watcher 也永不会被孤立。**关键推论:store 不在代码仓库里**,所以同一个 agent 能把它的记忆带进另一个仓库——`agit a track frontend && agit start`。

### ③ 绑定是提交进版本库的:队友克隆后即可取回
```sh
cat .agit.toml
```
```text
version = 1

[[agent]]
id     = "agt_3f17824b-0bba-74c2-804e-6ccb4c6f8bd0"
name   = "ratelimit"

[[agent]]
id     = "agt_493a5253-8c04-75e9-8a3b-39ce9e0b2266"
name   = "identity"

[defaults]
agent = "ratelimit"
```
> 讲解:这是协作得以成立的那一块。`.agit.toml` **提交在代码仓库根部**,告诉任何一个克隆者:这个仓库用哪些 agent、它们的身份是什么、去哪里取。队友克隆后一句 `agit a track ratelimit` 就把记忆拉了下来。其中的 `id` 是完整性校验:如果 hub 上的 `ratelimit.git` 被重建过,或 DNS 被挪走,按名字/URL 认人会把你静默绑到**另一个** agent 上;带上 aid,agit 会直接拒绝。

# 第二幕:原始 session 已被版本化,且密钥不会外泄

### ① 会话收录在 agent 自己的 store 里
```sh
agit a log --oneline
```
```text
d4c174d ratelimit: its session on this codebase
b6d98a2 agit: mint agent ratelimit (agt_3f17824b-0bba-74c2-804e-6ccb4c6f8bd0)
```
> 讲解:claude-code 与 codex 本身会将完整会话(转录 jsonl、子 agent、工具结果)转储到磁盘,`agit snap` 将该转储镜像进**所属 agent** 的 store,其中不含 fact、证据或 schema,仅为原始会话。两个运行时是对等的,没有默认值。`agit a <git 命令>` 就是在这个 store 上跑 git——`agit a log`、`agit a commit`、`agit a push` 都是原样的 git。

### ② 转储完整会话意味着可能夹带密钥,故提交时会执行扫描
```sh
S=$(agit a info ratelimit | awk '/^store/{print $2}')
echo '{"type":"user","message":{"content":"AKIAIOSFODNN7EXAMPLE"}}' > "$S/sessions/claude-code/LEAK.jsonl"
agit a add -A && agit a commit -m leak
```
```text
Found suspected secrets:
  sessions/claude-code/LEAK.jsonl:1  [aws-access-key-id]  AKIA…******

1 of them. Once the AgentState is pushed, a teammate who pulls carries them along.
Fix it. Or use --no-verify to bypass this hook and explicitly own the consequences.
```
> 讲解:由于转储的是完整会话,agent 读取过的 `.env` 或打印过的连接串均会留在转录中。pre-commit 钩子扫描到后即拦截本次提交,阻止其被 push 至团队。这道闸门在 **store 被创建时**就装上,无论它从哪扇门来(`a new`、`a track`、`a import`、`init`)——一个漏装了钩子的 store 会**静默地**什么都不扫。扫描采用高精度规则,故 UUID、requestId 等高熵噪声不会误报,仅命中真实密钥。
```sh
rm "$S/sessions/claude-code/LEAK.jsonl" && agit a add -A
```

# 第三幕:两个 agent 在两条分支上产生分叉

```sh
git log --oneline feature-a feature-b
git diff feature-a feature-b -- src/auth.js
```
```text
feature-a: login rate limiter on user_id
feature-b: rename user_id -> uid
payments: base
```
> 讲解:两条分支源自同一起点。`ratelimit` 的 `loginKey()` 读取 `req.body.user_id`,`identity` 已将该字段重命名为 `uid`。git 会在无冲突标记的情况下将两行文本合并,并引入一处静默缺陷。文本合并无法察觉此问题,须由语义合并处理。注意:**分叉发生在两段记忆之间,而不是两个分支名之间**——代码分支只是会话上记下的一条注记。

# 第四幕:`merge`,两个 agent 通过对话厘清差异

```sh
agit a merge identity
```
```text
identity is a different agent (agt_493a5253…) — reconciling by dialogue; histories stay separate.
Two worktrees: A@feature-a · B@feature-b (each carries its own diff since the common ancestor).
Reviving both sessions (read-only): A=1b229eb1 (1 local) … B=8eb16cfc (1 incoming) …

A → Since the fork I added one line to src/auth.js: loginKey(req) returning req.body.user_id,
    used as a rate-limit bucket key. What did you change?

B → My only change is that field rename: getUser now returns req.body.uid, no loginKey anywhere.
    CONFLICT: I renamed user_id -> uid, so your loginKey returns req.body.user_id = undefined once
    callers send the new payload — every request buckets under the same undefined key. DONE

A → Confirmed against my tree: grep finds only src/auth.js:1-2, no other callers. Your read is exact —
    git auto-merges the two lines textually and ships the break silently. DONE

── merge result ──
Agreed:
- Only conflict: identity renamed user_id → uid (src/auth.js:1); ratelimit's loginKey (src/auth.js:2)
  still reads user_id.
- Post-merge loginKey returns undefined for every request → all callers collapse into one bucket.
- Keep the rename; loginKey moves to req.body.uid, in the same commit as the rename.

Transcript archived → <store>/sessions/sync/1b229eb1-8eb16cfc.md
Merged state (both contexts + the reconciliation) — resume it to continue:
  (cd /tmp/agit-demo/showcase && claude --resume 1b229eb1-2e17-7e91-82a6-a2236d52340b)

Nothing left for you to decide.
```
> 讲解:此为全场核心。请先看**第一行**:模式是由**身份**决定的,而不是由 git 历史猜的。`identity` 的 aid 与我不同 ⇒ 它是**另一个 agent** ⇒ 只做对话式对齐,两份记忆都完整保留(这正是 PRD 的「两个 agent 轻量同步上下文,然后你自己接着合并代码」)。若目标的 aid 与我**相同**(比如队友 push 上来的同一个 agent 的另一份副本),则在对话之后连历史一起 merge,重新变回一段记忆。
>
> `merge` 不做摘要,而是将两条真实会话同时复活(只读),各自运行于其所属分支的 worktree 中,并将「自公共祖先以来的 diff」作为 ground truth 传入。两个 agent 通过阅读代码厘清差异。由于双方均携带完整上下文,交叉冲突得以浮现,而一次性摘要合并会将其遗漏。产物为一个可 resume 的合并后会话(而非 `CLAUDE.md`):resume 该会话,即可携带两个 agent 已对齐的上下文继续工作。当真实冲突无法达成一致时,`merge` 会中止并请求人工裁决,并将裁决结果写入该合并会话。

```sh
agit a list                     # 两个 agent 都还在,各自完整
agit a log --oneline            # 合并对话本身亦已版本化
sed -n '1,20p' "$S"/sessions/sync/*.md
```
> 讲解:对话本身归档于 `sessions/sync/` 目录,作为记录两个 agent 如何对齐的版本化 provenance,而不仅是结论。两个 agent 事后依然各自完整——这是「不同 agent」这一模式的定义。

# 本场演示体现的能力

| 能力 | 对应步骤 |
|---|---|
| agent 即记忆:以身份为键,独立于任何一个代码仓库 | 第一幕 ①② |
| 跨端迁移得以可能:store 不在代码仓库里,`agit a track` + `agit start` 即可带走 | 第一幕 ② |
| 提交式绑定:队友克隆代码仓库即知道用哪些 agent,并可取回记忆 | 第一幕 ③ |
| 身份完整性校验:remote 被重建也不会静默绑到另一个 agent | 第一幕 ③ |
| 最小侵入:直接版本化运行时转储的会话,不定义 fact 或 schema | 第二幕 ① |
| store 即 git 仓库:push、pull、clone 无需额外实现 | 第二幕 ① |
| 密钥防线:转储完整会话亦不外泄密钥,且不被 UUID 噪声淹没 | 第二幕 ② |
| 由身份而非猜测决定合并模式:不同 agent ⇒ 两份记忆都保留 | 第四幕 |
| agent 驱动的语义合并:复活两侧,通过阅读代码厘清差异 | 第四幕 |
| 仅真实冲突请求裁决:可自动合并者均已合并 | 第四幕 |
| 可 resume 的合并态:resume 会话,携带双方上下文继续 | 第四幕 |
| 两个运行时对等(claude-code、codex),无默认值 | 第二幕 ① |
| 更换 LLM 后端:`AGIT_LLM=codex` 或 `AGIT_LLM_CMD=…` | 无 |
