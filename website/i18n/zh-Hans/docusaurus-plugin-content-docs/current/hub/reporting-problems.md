# 报告问题

当中枢出现故障时，有一个标识符能让它变得可被查找：请求 id。每一个中枢响应都携带一个 `X-Request-Id` 头部，而每一个 JSON 错误体都会把同一个 id 折入一个 `request_id` 字段中。在报告中附上它，运营者无需任何对你账户或会话的访问权限，就能在服务器日志中找到对应的条目。

## 请求 id

中枢为每一个请求生成一个全新的 16 字符 id，并将其盖印在响应上，无论成功还是出错。客户端提供的 id 会被忽略；服务器始终生成自己的 id。该 id 在服务器日志中端到端地串起整个请求（认证事件、处理器、计时），因此它是最值得引用的单一信息。

你会在以下位置找到它：

- 任意请求上的 `X-Request-Id` 响应头部，以及
- JSON 错误体（状态码 400 及以上）内部的 `request_id` 字段。

## 从 Web 界面报告问题

Web 界面的 **Report a problem** 会收集运营者所需的上下文，且不含任何敏感信息：

- 你的浏览器与当前页面，
- 近期失败的请求，每个都带有其请求 id，以及
- 任何控制台错误。

复制或下载该报告并发送它。由于它已经携带了请求 id，运营者可以直接把它与服务器日志相关联。

## 报告失败的推送或拉取

从客户端失败的推送或拉取是一个中枢连通性问题，`agit debug` 会将其捕获：其 `connectivity.txt` 部分会记录中枢返回的请求 id。用 `agit debug` 收集一个诊断包（它会写出一个带时间戳的目录；用 `--out <dir>` 可指定你自己的目录），查看它（在写出之前一切都已通过密钥扫描器（secret scanner）脱敏，且不会上传任何内容），并附上该目录。参见 [诊断](../cli/diagnostics.md)。

## 发往何处

在 [github.com/Einsia/agent-git/issues](https://github.com/Einsia/agent-git/issues) 提交一个 issue。对于中枢问题，请附上 Report a problem 的内容与请求 id；这一对信息通常足以定位故障。一条一行的建议同样受欢迎，且无需是一份完整的报告。

如果你自行运行中枢，请求 id 同样是进入你自己日志与指标的钥匙；参见 [运维](../self-hosting/operations.md)。
