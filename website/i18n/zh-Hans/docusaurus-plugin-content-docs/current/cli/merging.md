# 合并会话

当两个人运行同一个智能体并各自 push 时，他们的会话会分叉。`agit a merge` 将它们调和。它并不逐行 diff 转录记录。
取而代之，它以只读方式复活双方各自最新的会话，让它们对照当前代码比对彼此的工作，产出一个可恢复的合并会话，外加一份
真正冲突的清单。合并在本地运行。

`sync` 作为 `merge` 的别名保留，受相同的作用域把关，因此 `agit a sync` 仍然有效，而代码仓库中的 `agit sync` 会
透传给 git。

## 分叉何时发生，以及 pull 如何将其显露

你通常从 `agit a pull` 走到一次合并。在分叉时，一次裸 `git pull` 会对 JSONL 转录记录做文本合并，而这恰恰会损坏
合并本应调和的那些会话。因此 `agit a pull` 在能安全快进时快进（`--ff-only`），而当两段历史已分叉时，它拒绝并把你
指向针对已分叉上游引用的合并命令：

```
diverged: local and remote both added sessions (a text merge corrupts transcripts)
  agit a merge origin/main   reconcile by dialogue
```

## 运行一次合并

```bash
agit a merge frontend
```

目标是另一个智能体（一个智能体名称）或存储库中的一个 git 引用，绝不是代码分支。裸名称经由[绑定](../get-started/concepts.md)
解析；一个引用则调和到同一存储库中较早的一点。当一个词同时命名一个智能体和一个引用时，用 `--agent X` / `--ref X`
可无需提示地消歧。

## 一次合并产出什么

每一侧最近的会话都以只读方式在各自的工作树中复活，携带它自两段历史的共同祖先以来引入的 diff。两个会话交换摘要，
并通过阅读当前代码调和它们所能调和的部分。它们无法调和的部分（一方所做、而另一方的改动与之矛盾的某个决定）会作为
一处冲突显露出来，交由你裁定。

由于双方都作为真实会话复活，运行时 CLI（`claude` 或 `codex`）必须已安装。由哪个运行时复活它们，遵循惯常规则：你用
`--from` 指名的那个，否则是唯一在场的那个，再否则 agit 会询问。参见[运行时](./runtimes.md)。

产出是一个你可以像其他任何会话一样恢复的会话。一次成功的合并会把该会话自动快照进存储库，作为该智能体的最新会话，
于是一次裸 `agit start` 或 `agit resume` 就会取到它。你无需手动快照它。若一个密钥溜进了合并后的转录记录，密钥关卡
会把它挡在历史之外，并使合并以非零退出，与一次快照的行为完全一致。参见[密钥](./secrets.md)。

## 同一智能体对不同智能体（aid 感知）

目标的 aid 决定合并走多远：

| 目标 | 被调和的方式 | Git 历史 |
|---|---|---|
| 相同 aid（同一个智能体，例如队友 push 的副本） | 通过对话 | 合并为一段历史 |
| 不同 aid（一个各异的智能体） | 通过对话 | 保持不变；各自保留自己的历史 |

按 aid 合并，正是为什么队友拉取你的会话、而你拉取他们的会话时，双方会汇聚到一段历史；而合并两个恰好同名的智能体，
则不会把它们叠合在一起。身份是 aid，而非名称。参见[核心概念](../get-started/concepts.md)与[分叉](./divergence.md)。

## 冲突账本

当一次合并显露出冲突时，你逐一裁定：接受对话所命名的某个选项、键入你自己的决定，或将其保持敞开。那些决定被写入合并
转录记录旁的一个账本，位于 `<agent>/sessions/sync/<a>-<b>.decisions.md`，于是已了结与已延后的内容都留存于存储库中，
而不仅在恢复出的会话内部。已了结的决定还会被折叠进合并后的会话，因此当你恢复它时，智能体会带着这些已经定下的决定
继续。

## 综合经由一个模型进行

冲突综合经由一个模型进行，由 `AGIT_LLM` 选定（参见[配置](./configuration.md)）。若无可用模型，`agit a merge` 会
列出未决冲突而不去解决它们，而其他每一步仍然运行。由于结果由一个模型产出，结果并非确定性的：同一次合并跑两遍，合并
后的会话可能不同。双方的原始会话始终被提交并版本化，因此它们仍是事实来源。一次你不满意的合并，就是一次你可以丢弃
并重做的合并。

## 选项

| 选项 | 效果 |
|---|---|
| `--from <runtime>` | 当两个运行时都在场时，选择由哪个复活会话。 |
| `--both` | 将合并后的会话写到两条分支上，而非一条。 |
| `--quick` | 跳过摘要交换，仅从 diff 出发调和。更快，但不那么彻底。 |
| `--splice` | 无需模型，将双方合并进一个会话。见下文。 |
| `--dry-run`（别名 `--preview`） | 显示合并会做什么，但不实际运行。见下文。 |

## 用 `--dry-run` 预览

```bash
agit a merge frontend --dry-run
```

`--dry-run` 只运行真实合并起始时的那些检视阶段（解析目标、读取每一侧的会话、为每一侧挑选发声会话），并打印计划：
目标、模式、每一侧各有多少会话，以及为每一侧挑选的发声会话。它到此为止。不运行任何模型、不复活任何会话、不写入任何
内容，因此它既不需要 `AGIT_LLM` 也不需要运行时 CLI。去掉 `--dry-run` 重跑即可执行它。

## 用 `--splice` 合并双方

```bash
agit a merge peer --splice
```

`--splice` 完全跳过模型。它不去调和双方，而是把它们合并起来：它取 A 的完整转录记录，从两侧分叉之点起追加 B 的尾部，
将会话 id 与工作目录规整到一个之上，并把结果作为一个单一的新会话安装。什么都不被调和，也不计算冲突；你恢复这个合并
后的会话，智能体便在一个窗口中读到双方的上下文。由于它不运行任何模型、也不复活任何东西，`--splice` 既不需要
`AGIT_LLM` 也不需要运行时 CLI，且它是确定性的。相同 aid 的目标随后仍会合并 git 历史。当你宁愿把整幅图景交给一个
全新的智能体、而非让两个复活的智能体协商时，或当没有配置任何模型时，使用它。
