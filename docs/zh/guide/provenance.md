# 验证会话的产生者

来源溯源把一次捕获到的会话，与产生它的机器、以及持有那台机器密钥的人绑定在一起。每台机器都会对它所捕获的会话签名。要确认某次会话是某个特定的人产生的，就把那台机器的密钥注册到此人的 hub 账户下，并对照 hub 进行验证。

## 设置 git 身份

会话归属到的人，就是您的 git 提交者身份，按 git 解析它的方式解析（先看本地配置，再看全局配置）。agit 不会替您编造。像对待任何 git 仓库那样，设置一次即可：

```
git config --global user.email you@example.com
git config --global user.name  "Your Name"
```

您也可以用 `agit a config user.email you@example.com` 逐 agent 设置，那不过是对存储库执行普通的 git。身份未设置时，agit 会拒绝记录会话，因为会话正是被归属的那个对象。agit 自己的簿记提交（创建 agent、重命名）由 `agit@local` 署名，从不需要您的身份，因此一台新机器可以在您设置身份之前先创建 agent。

## 显示本机的签名密钥

每台机器都有一把 ed25519 签名密钥，在首次使用时创建，存放于 `$AGIT_HOME/identity/ed25519`（私钥权限为 `0600`）。显示它的公开身份：

```
agit provenance key
```

当 agit 捕获一次会话时，它会用那把密钥签名，并把签名记录进会话已提交的附属文件里，与会话记录摘要、agent 的 aid、您的提交者邮箱以及开始时间放在一起。公钥可以安全地分享；签名随会话保存在存储库中。

## 验证一次会话

```
agit provenance verify <session>
```

参数是解析出的 agent 的存储库中的会话路径或 id。验证会重新计算会话记录摘要、重建被签名的消息，并用记录所携带的公钥核对签名。它会报告以下之一：

| 结果 | 含义 | 退出码 |
|---|---|---|
| verified | 内容完好，且签名与其所记录的密钥相符。 | 0 |
| unverified, no signature | 会话不带签名（在签名机制出现之前捕获，或当时没有可用的密钥）。 | 0 |
| tampered | 会话记录在被签名之后发生了改动：它当前的摘要与被签名时的摘要不符。 | 非零 |
| bad signature | 存在签名，但无法用其所记录的密钥通过验证。 | 非零 |

未签名的会话报告 “unverified”，且从不阻断。存在但核对不通过的签名，则是一次硬失败，以非零码退出。

自我验证证明内容完好、签名与所记录的密钥相符。它并不说明那把密钥属于谁，这正是为什么结论是 “verified” 而绝不是 “trusted”。

## 以“人”为单位验证

要把“由这把密钥签名”变成“由这个人签名”，需要把机器的密钥注册到一个 hub 账户下。此后 `agit provenance verify` 会在给出结论之前，先到 hub 的注册表里查找提交者邮箱。

1. 设置您的 git 身份，使 snap 下来的会话带上您的邮箱（见上文）。
2. snap 一次会话，或让守护进程捕获它。它的来源信息现在就把您的邮箱与这台机器的签名密钥绑定在了一起。
3. 把这台设备的密钥注册到您的 hub 账户下：

   ```
   agit identity register <you>
   ```

   这会打印出一段可粘贴的注册块。把它粘贴进 hub 网页界面，即可把这把密钥登记到您的账户名下。`agit identity show` 会报告这台机器已登记了什么。
4. 验证：

   ```
   agit provenance verify <session>
   ```

   当您的邮箱映射到某个 hub 账户，而该账户已注册的密钥中包含会话的签名密钥时，结论就升级为 VERIFIED AS 您。

带注册表的结论：

| 结果 | 含义 | 退出码 |
|---|---|---|
| VERIFIED AS `<user>` | 自我验证通过，且提交者邮箱映射到某个 hub 账户，其已注册密钥中包含该签名密钥。这是唯一“以人为单位验证通过”的结论。 | 0 |
| signed, unregistered | 自我验证通过，但提交者邮箱映射不到任何账户（或没有可达的 hub）。只归属到一把密钥，尚未归属到一个人。 | 0 |
| KEY MISMATCH | 自我验证通过，邮箱也映射到某个账户，但签名密钥不属于该账户已注册的任何一把密钥：可能是伪造。 | 非零 |

已注册的密钥集合在首次看到时就被钉定，因此 hub 事后无法换入一把密钥来制造虚假的“verified as”。一个错误或伪冒的提交者邮箱绝不会产生虚假的 VERIFIED AS：它会退化为 “signed, unregistered” 或 KEY MISMATCH。针对一把已注册密钥的签名才是真正的边界；邮箱只是查找用的把手。
