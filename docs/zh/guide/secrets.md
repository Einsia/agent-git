# 不让密钥进入共享历史

会话是 agent 读取和运行过什么的记录，因此它可能带有 agent 碰巧看到的密钥：它打开过的某个 `.env`、它打印出来的某个令牌。agit 会在提交或推送任何内容之前扫描密钥，这样记录里的密钥就不会不小心进入存储库，也不会不小心到达团队手中。

## 扫描在何时运行

agit 在自己的入口处于进程内扫描，早于它把工作交给 git：

- `agit a commit` 扫描暂存区（即这次提交将写入的内容）。
- `agit a push` 扫描即将发布的提交范围。
- 守护进程或 `agit snap` 拍下的每一份快照，都以同样方式扫描。

只有扫描通过，提交或推送才会进行。`agit init` 还会在存储库上安装 pre-commit 和 pre-push 钩子，这样一次裸的 `git commit` 或 `git push` 也会被扫描。agit 自己的提交和推送无法用 `git --no-verify` 蒙混放行：它们先在包装层里扫描，再把 `--no-verify` 传给它所发起的 git 调用，因此即便 git 自己的钩子被跳过，包装层的扫描也始终会运行。

随时可以手动扫描 agent 的会话：

```
agit a scan
```

## 扫描发现问题时

扫描会打印每一处疑似密钥（文件、行、规则和一段摘录）并拒绝放行。被阻止的提交不会产生任何提交；被阻止的推送什么也不发送。您有三条前进的路。

### 移除密钥

把密钥从会话中删掉，再试一次。

### 标记误报

如果某处命中是已知安全的（文档示例、占位符），就把它豁免：

- 在那一行加上 `agit:allow-secret` 行内指令。该指令只豁免一个物理行，并随提交一同保存。
- 或者把该字符串加入存储库的 `.agit-allow-secrets` 白名单文件，它会豁免一个匹配到的字符串。

hub 对这两种方式都认可，因此这样标记的误报既能通过本地扫描，也能通过服务器端扫描。

### 覆盖本地扫描

```
AGIT_ALLOW_SECRETS=1 agit a push
```

`AGIT_ALLOW_SECRETS`（也接受 `true` 或 `yes`）会让操作在有发现的情况下仍然放行。这是一次可见、可审计的覆盖：agit 每次采纳它时都会披露，指明是哪个操作、有哪些发现。这正是它与 `git --no-verify` 的区别 —— 后者不留任何痕迹。该覆盖只清除 agit 的本地扫描。推送到某个 hub 时，仍要面对 hub 自己的扫描，而这个标志够不到那里。

## hub 会再扫描一次

当您推送到某个 [hub](../hub.html) 时，它会在进入时于服务器端扫描，并拒绝携带密钥的推送。这道兜底即便对从未运行本地扫描的客户端、或用 `AGIT_ALLOW_SECRETS` 清除了扫描的客户端也依然有效，因为那个标志是客户端一侧的，不会传到服务器。一处真正的误报，绕过 hub 扫描的方式与绕过本地扫描的方式相同：`agit:allow-secret` 行内指令随提交一同带过去，或者由运维者把该字符串加入裸仓库的 `.agit-allow-secrets`。

密钥扫描识别已知的密钥形态。它是一道有力的兜底，而非对某种新奇密钥的保证。
