# 协调分叉的会话

当两个人运行同一个 agent 并且都推送时，他们的会话就会分叉。`agit a merge` 负责协调它们。它并不逐行比对会话记录，而是把两边最新的会话以只读方式复活，让它们各自把自己的工作与当前代码相比对，最终产出一次可恢复的合并会话，外加一份真正的冲突清单。

您通常是从 `agit a pull` 走到合并这一步的：当同事的会话能快进您的会话时，pull 会直接应用它们；当两条历史已经分叉时，它会停下并把您送到这里。

```
agit a merge frontend
```

目标可以是另一个 agent（一个 agent 名称），也可以是存储库里的某个 git ref。裸名称通过[绑定](concepts.html)解析；ref 则针对同一存储库中更早的某个点进行合并。目标绝不会是代码分支。

## 合并会产出什么

每一边最新的会话都会在各自的工作区里以只读方式复活，带着它自两条历史的共同祖先以来所引入的差异。两次会话交换各自的摘要，并通过阅读当前代码尽力协调彼此。协调不了的部分（一边做出的、与另一边改动相冲突的决定）会作为冲突呈现出来，交由您裁定。

因为两边都是作为真正的会话复活的，所以运行时 CLI（`claude` 或 `codex`）必须已安装。由哪个运行时复活它们，遵循一贯的规则：您用 `--from` 指明的那个；否则是唯一存在的那个；再否则由 agit 询问（参见[在运行时之间迁移会话](runtimes.html)）。

产出是一次像其他会话一样可供您恢复的会话：

```
claude --resume <id>
codex exec resume <id>
```

一次成功的合并会把该会话作为 agent 的最新会话记录进存储库，因此一句裸的 `agit start` 或 `agit resume` 就能接上它。您无需手动 snap。如果有密钥混进了合并后的记录，密钥关卡会把它挡在历史之外，合并以非零状态退出 —— 与 snap 的行为完全一致（参见[不让密钥进入共享历史](secrets.html)）。

## 同一个 agent 还是不同的 agent

目标的 aid 决定合并走多远：

| 目标 | 协调方式 | git 历史 |
|---|---|---|
| 相同 aid（同一个 agent，例如同事推送的副本） | 通过对话协调 | 合并为一条历史 |
| 不同 aid（一个各自独立的 agent） | 通过对话协调 | 各自保留，互不合并 |

正是因为按 aid 合并，同事拉取您的会话、您拉取同事的会话，才会收敛到同一条历史；而合并两个碰巧同名的 agent 并不会把它们揉到一起。身份是 aid，不是名称。参见[概念](concepts.html)。

## 冲突记录簿

当合并暴露出冲突时，您逐一裁定：接受对话提出的某个选项、输入您自己的决定，或暂且搁置。这些决定会写入合并记录旁边的一份记录簿，位于 `<agent>/sessions/sync/<a>-<b>.decisions.md`，因此哪些已经定夺、哪些是有意推迟，都保存在存储库里，而不仅仅存在于恢复出来的那次会话中。已定夺的决定还会被并入合并后的会话，因此当您恢复它时，agent 会带着这些已定的决定继续。

## 合并要经过一个模型

冲突的综合处理会经过一个模型，由 `AGIT_LLM` 选定（参见[配置](configuration.html)）。如果没有可用的模型，`agit a merge` 会改为列出未决冲突而不去解决它们，其余每一步照常运行。

因为结果是由模型产出的，所以结果并非确定性的：同一次合并跑两遍，得到的合并会话可能不同。两边的原始会话仍然被提交并版本化地保存在存储库里，因此它们始终是事实的来源。一次您不满意的合并，可以丢弃后重做。

## 选项

| 选项 | 结果 |
|---|---|
| `--from <runtime>` | 当两边都有会话时，指定由哪个运行时复活会话。 |
| `--both` | 把合并后的会话写到两个分支上，而非只写一个。 |
| `--quick` | 跳过摘要交换，仅凭差异进行协调。更快，但不够周全。 |
| `--splice` | 不经模型，直接把两边合并成一次会话。见下文。 |
| `--dry-run`（别名 `--preview`） | 显示合并将会做什么，但并不真正执行。见下文。 |

## 用 `--dry-run` 预览合并

```
agit a merge frontend --dry-run
```

`--dry-run`（别名 `--preview`）只运行真正合并开头的那些检查阶段（解析目标、读取每一边的会话、为每一边挑选发声会话），并打印出计划：目标、模式（同一个 agent，先协调再合并 git 历史；还是不同的 agent，只通过对话协调）、每一边有多少次会话，以及为每一边挑选的发声会话。它到此为止。不运行模型，不复活任何会话，也不写任何记录或记录簿，因此它既不需要 `AGIT_LLM` 也不需要运行时 CLI。去掉 `--dry-run` 重新运行即可真正执行。

## 用 `--splice` 合并两边

```
agit a merge peer --splice
```

`--splice` 完全跳过模型。它不去协调两边，而是把它们拼接起来：取 A 的完整记录，从两者分叉的那一点起接上 B 的尾段，把会话 id 和工作目录归一到同一个，并把结果安装为一次全新的会话。什么都不协调，也不计算任何冲突。您恢复这次拼接后的会话，agent 便在同一个窗口里读到两边的上下文，并从那里继续。

因为它不运行模型、不复活任何东西，所以 `--splice` 既不需要 `AGIT_LLM` 也不需要运行时 CLI，而且它是确定性的：同样的两次会话总是拼接出同样的字节。若目标为相同 aid，之后仍会像普通合并那样合并 git 历史。当您宁愿把全局交给一个全新的 agent，也不想让两个复活的 agent 彼此协商时，或者当没有配置模型时，就用它。
