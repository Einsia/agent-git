## 1. What problem it solves

Your code is public to the team through Git, but the agent's work stays trapped in a private session — what it
read, the judgments it made, what it planned next. Nobody else can see it or reuse it.

agit manages the agent's **raw sessions** directly. Claude Code (and Codex) already dump the entire session to
disk; agit versions that dump and syncs it with `push`/`pull`. When two branches diverge, the merge is done by
**reviving both agents and letting them talk it out** — reading each other's work and the code — and only real
conflicts stop to ask you.

> An earlier model ("hand-authored facts + evidence + deterministic merge") has been retired. For why, and the
> tradeoffs of the old vs. new approach, see [`风险分析.md`](风险分析.md).

## 2. Two repositories and an agent-driven merge

```
your-project/
├── models/user.ts            ← Environment: your code, untouched by agit
├── .gitignore                ← .agit/ is added automatically
└── .agit/
    ├── agent/                ← Agent Store: an independent git repository
    │   ├── agent.toml
    │   ├── sessions/claude-code/
    │   │   ├── <id>.jsonl        full transcript
    │   │   └── <id>/subagents, tool-results
    │   ├── sessions/codex/      Codex sessions (same layout)
    │   └── sessions/sync/       merge dialogue transcripts (provenance)
    └── workspace/            ← WorkspaceRevision: the Agent↔Environment pairing
```

### Two scopes

```
agit <git command>       →  operates transparently on your code repository (Environment)
agit -a <git command>    →  operates isomorphically on the Agent Store
```

> **⚠️ Ambiguity (worth memorizing):** `agit -a commit` (agent repo) vs. `agit commit -a` (code repo, where `-a`
> is git commit's own flag). The scope switch only recognizes the first token immediately after `agit`.

The Agent Store is just a git repository, so `push` / `pull` / `clone` / `fetch` / `log` / `diff` all pass
straight through for free.

## 3. Install

```bash
./build.sh --release              # why not cargo build: see the root README
cp target/release/agit ~/.local/bin/

cd your-project
agit init                         # create the Agent Store; re-run after a clone
```

> **Re-run `agit init` after a clone.** The reasoning is the same as for `.gitignore`/hooks: hooks don't travel
> with a clone.

## 4. Five-minute quickstart

```bash
cd your-project && agit init
agit -a snap                                    # mirror this project's Claude session dump
agit -a add -A && agit -a commit -m 'record the session'
agit -a push                                    # publish (secrets are scanned before push)

# on a teammate's machine
agit clone <url> && agit -a fetch origin
agit -a sync origin/main                        # agents reconcile by dialogue; only real conflicts ask you
```

## 5. Command reference

### Native verbs

| Command | What it does |
|---|---|
| `agit init` | Create the Agent Store + secret hook |
| `agit -a snap [--from <rt>] [--watch]` | Mirror the runtime's session dump into the Agent Store (default `claude-code`; `agit -a snap codex` is shorthand for the runtime). `--watch` = fully automatic: auto-mirror + auto-commit on every change. Formerly named `sync`. |
| `agit resume <src> [--as <rt>] [--exec]` | Load a session into a runtime and continue (the universal loader; `--as` converts across runtimes, `--exec` launches it) |
| `agit graph` | Show the Workspace-State timeline and relation edges |
| `agit -a sync <ref> [--both] [--from <rt>]` | Reconcile this branch's agent with `<ref>`'s agent **by a live dialogue**: revive both agents read-only, let them converse and resolve by reading code, surface only real conflicts, and leave a **resumable merged session**. Works for `claude-code` and `codex`. |
| `agit convert <src> --to <rt> [--from <rt>] [--cwd <path>] [--write]` | Convert a session so another runtime can resume it (dry-run by default; `--write` persists it and prints the resume command) |
| `agit clone <url>` | Pull the team Agent Store in one command (clone + install hooks) |
| `agit -a scan [--staged]` | Scan the session dump for secrets |
| `agit workspace [log]` | Show the Agent↔Environment pairing |
| `agit workspace restore [N]` | Roll both repositories back together to a pairing's joint state (the undo for the joint version history) |
| `agit adapter` | List the runtime adapters (`claude-code` and `codex`, both implemented) |

### Passthrough (identical to git)

`add` / `commit` / `status` / `log` / `diff` / `branch` / `checkout` / `merge` / `push` /
`pull` / `fetch` / `clone` … with no prefix they operate on the code repository; with `-a` they operate on the
Agent Store.

`agit` **does not replace** `git`; the two coexist.

## 6. sync: how the merge happens

Sessions are files with distinct UUIDs, so when `git pull`/`fetch` brings in the peer's sessions there is **no
text conflict** — they simply land side by side under `sessions/`. The real "merge" is semantic, and `sync` does
it by putting the two agents in dialogue:

```bash
agit -a fetch origin
agit -a sync origin/main
```

1. A three-dot diff (`HEAD...<ref>`) finds the sessions the peer added since the common ancestor — the divergent
   tail. If `<ref>` added nothing, there is nothing to sync.
2. It picks one representative "latest" session per side (A = your latest local session; B = the peer's latest
   incoming session) and **revives both** as fresh, read-only sessions through the convert machinery, so your real
   sessions are never touched.
3. **Two worktrees:** each agent runs in its own branch's checked-out tree, carrying its own diff since the
   common ancestor as ground truth. The agents get read-only tools only (`Read`/`Grep`/`Glob`) — they resolve by
   reading code, not by editing it. (If both code branches aren't present in the repo, it falls back to a single
   tree and says so.)
4. The two agents **converse** over the divergent tail until they've reconciled or hit a conflict that needs a
   human. Because each side reasons from its full context, cross-cutting conflicts (e.g. B renamed a field that A's
   code still buckets on) surface that a one-shot summary merge would miss.
5. **Output is a resumable merged session, not a `CLAUDE.md`.** The A-side session now contains the whole
   reconciliation — that *is* the merged state. `sync` prints the command to resume it:

   ```
   Merged state (both contexts + the reconciliation) — resume it to continue:
     (cd your-project && claude --resume <id>)
   ```

   The dialogue transcript is also archived under `sessions/sync/` for provenance. With `--both`, a B-side merged
   session is emitted too, so the peer's branch can resume with the combined context as well.
6. **Only real conflicts stop to ask you.** At a terminal, `sync` walks you through each open conflict and bakes
   your decisions into the merged session. Run non-interactively, it prints what needs deciding and exits non-zero
   instead of prompting.

**Requirements and current limits:**

- `sync` needs the runtime CLI (`claude` or `codex`) on the machine, because both sides of the dialogue are real
  resumed sessions. There is no offline mechanical-merge fallback (that was the old `reconcile` command, now
  removed).
- MVP scope: one "latest" session per side; both sides `claude-code` by default. Use `--from codex` to run the
  dialogue with Codex sessions. Multi-session sides and mixed-runtime dialogue are follow-ups.

> **The dialogue/synthesis step is non-deterministic** (there's a model in the loop), so the same inputs can
> produce different results on two runs. That's a deliberate tradeoff. Where it can go wrong — quietly merging
> incorrectly, non-reproducibility, prompt injection — is covered in [`风险分析.md`](风险分析.md); worth reading
> before you rely on it.

> **Design direction (context, not all shipped):** the broader primitive set being designed is
> `snap` / `resume` / `sync` / `push`-`pull`-`clone` / `graph`, where `resume` becomes a first-class verb and the
> merge output is always a state you resume. Today you resume the merged session directly with the runtime CLI
> (`claude --resume` / `codex exec resume`). See
> [`docs/plans/2026-07-16-workspace-state-primitives-design.md`](plans/2026-07-16-workspace-state-primitives-design.md).

## 7. Swapping the LLM backend

The dialogue itself runs through the real resumed runtime CLI. The **synthesis** step — turning the dialogue into
an "Agreed" summary plus the list of open conflicts — goes through `src/llm.rs`, which is pluggable:

```sh
export AGIT_LLM=claude               # default, local `claude -p`
export AGIT_LLM=codex                # local `codex exec` (implemented; uses codex exec -o <file> for a clean reply)
export AGIT_LLM_CMD="<any command that reads a prompt from stdin>"   # overrides everything, e.g. ollama
```

If no LLM backend is available, `sync` still runs the dialogue and surfaces the open items, but skips the
synthesized "Agreed" summary (it's marked `(no LLM backend, not synthesized)`).

**Codex is a first-class runtime** (not a stub): its session dumps (`~/.codex/sessions/`, filtered to this
project by `cwd`) can be captured with `agit -a snap codex`, run on either side of `agit -a sync --from codex`,
and converted with `agit convert`. Parsing lives in `src/adapter/codex.rs`.

## 8. Hub

Team-level hosting and browsing; see [`hub.md`](hub.md) for details. In short: the Hub is **Registry + Sync +
read-only rendering** and **does not merge** (merging happens locally with `agit -a sync`). The frontend is an
embedded React SPA (session spine / provenance / permalinks / revision diff). **Push requires a write token**
(`agit-hub token add … --write`; only a sha256 digest is stored), and reads require a token too under
`serve --private`.

## 9. Security and known limitations

- **`agit -a scan` / commit / push hooks** scan sessions for secrets. Dumping the whole session means any `.env`
  or connection string the agent `cat`ed is in the transcript, and would be shipped if unscanned. The session
  (jsonl) scan uses high-precision rules only, so it isn't tripped by high-entropy strings like UUIDs or
  requestIds — but it **cannot catch general sensitive content** (trade secrets, PII).
- **`sync` is non-deterministic**, depends on an external model, and feeds untrusted sessions to an LLM (a
  prompt-injection surface).
- **No automatic staleness detection** between a session's conclusions and the current code — a resumed session
  can be out of date relative to the repo.
- Details and mitigations for each of these are in [`风险分析.md`](风险分析.md).

## 10. End-to-end workflow example

```bash
# ── Alice ────────────────────────────────
cd payments && agit init
agit -a snap                                   # mirror her session
agit -a add -A && agit -a commit -m 'worked out the user model'
# The Hub admin issues a write token first: agit-hub token add alice --write (shown once)
agit -a remote add origin http://alice:<token>@hub:8177/payments.git
agit -a push -u origin main

# ── Lin (the next day) ────────────────────
cd payments
agit clone http://hub:8177/payments.git        # pull the team Agent Store
agit -a snap && agit -a add -A && agit -a commit -m 'my session'
agit -a fetch origin
agit -a sync origin/main                        # agents reconcile by dialogue; only real conflicts ask him
# sync prints a `claude --resume <id>` command — resume it to continue with the merged context
```
