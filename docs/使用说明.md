## 1. 它解决什么问题

代码通过 Git 对团队公开了，但 **Agent 的工作困在私人 session 里**——它读过什么、
得出什么结论、为什么这么判断、做到哪一步，别人看不到、也没法复用。

agit 把这些变成**可提交、可比较、可合并、可复用**的版本化对象。而且每条结论都**带证据出处**，
证据会随代码变化而**自动失效**——这是它和「记录状态」的工具（Claude Code 的 rewind、
Zed 的 checkpoint、Shepherd）的根本区别。

## 2. 核心概念：两个库 + 配对

Git 只版本化代码。agit 认为 Agent 是系统内持续积累状态的执行者，所以要版本化的对象扩展成：

```
AgentGit State = Agent State（agent 知道什么）
               + Environment State（基于哪个代码版本）
               + Relations（配对关系）
```

落到磁盘上是**两个 git 仓库 + 一个配对**：

```
你的项目/
├── models/user.ts            ← Environment：你的代码，agit 原封不动
├── .gitignore                ← 自动加入 .agit/
└── .agit/
    ├── agent/                ← Agent Store：独立 git 仓库，装 AgentState
    │   ├── agent.toml        ← Agent 身份
    │   └── state/
    │       ├── goals.md          目标
    │       ├── constraints.md     约束
    │       ├── progress.md        进度
    │       ├── artifacts.md       产物引用
    │       ├── facts/<主题>.md    事实 + 证据（一个 fact 一个文件）
    │       ├── _evidence_pool.md  证据候选池（抽取的原材料）
    │       └── _session.json      溯源
    └── workspace/            ← WorkspaceRevision：Agent↔Environment 配对
```

**没有隐藏数据库，没有 `~/.agit`（Hub 除外）。全部是 git 对象。**

### 两个 scope

```
agit <git 命令>       →  透明作用在你的代码仓库（Environment），行为和 git 一模一样
agit -a <git 命令>    →  同构作用在 Agent Store
```

比如 `agit log` 看代码历史，`agit -a log` 看 context 历史；`agit status` 看代码改动，
`agit -a status` 看 context 改动。整套 git 命令面两个库都能用。

> **⚠️ 一个容易踩的歧义（务必记住）**
> ```
> agit -a commit      →  agent scope（提交 context）
> agit commit -a      →  code scope（-a 是 git commit 的参数，提交所有代码改动）
> ```
> **scope 开关只认紧跟 agit 的第一个 token。** 子命令后面的 `-a` 原样交给 git。

### fact：一个结论一个文件，主题即路径

```markdown
---
subject: api/user/id-field-name       # 就是文件路径，也是合并时的对齐键
tier: reversible
author: alice
created: 2026-07-13T09:46:14Z
evidence:
- 'file:models/user.ts:4 #a937b4a5'   # 指向代码仓库；#... 是采集当时那几行的摘要
---

用户标识字段叫 user_id，不是 uid。
```

因为「主题即文件路径」，git 的三方合并**天然区分**两种情况：
- 两个 agent 改了**同一条结论** → 同一个文件 → git 报冲突，交给证据裁决
- 两个 agent 学到**不同的知识** → 不同文件 → git 静默合并，谁也不丢

## 3. 安装

```bash
# 在项目里
./build.sh --release              # 见下方说明
cp target/release/agit      ~/.local/bin/
cp target/release/agit-hub  ~/.local/bin/   # 需要 Hub 时
```

> **为什么是 `./build.sh` 而不是 `cargo build`**：依赖用了 edition2024，`Cargo.lock` 是 v4，
> 需要 cargo ≥ 1.78（Ubuntu 22.04 apt 自带的是 1.75）。`./build.sh` 会自动找到能用的 cargo
> （优先 `~/.cargo/bin/cargo`），不需要你改 PATH。

在你的代码仓库里初始化：

```bash
cd 你的项目
agit init         # 建 Agent Store（.agit/agent）+ 配对基建；把 .agit/ 加进 .gitignore
```

> **clone 之后要重跑 `agit init`。** `.gitattributes` 跟着 clone 走，但 merge driver 的配置
> 和 hook 不会（git 的安全设计，防止 clone 一个仓库就执行别人的命令）。

## 4. 五分钟上手

```bash
cd 你的项目 && agit init

# 手写一条带证据的结论（证据指向真实代码，否则拒绝入库）
agit -a new api/user/id -e file:models/user.ts:4 -m '用户标识字段叫 user_id。'

# 校验：证据还对得上吗
agit -a verify

# 提交 context（注意 -a 在前）
agit -a add -A
agit -a commit -m '记录用户模型结论'

# 改一行代码，再校验 —— 那条结论会自动变 STALE
sed -i 's/user_id/userId/' models/user.ts
agit -a verify        # → STALE，退出码非零

# 看某条结论的来龙去脉
agit -a why api/user/id
```

## 5. 命令详解

### 透传（和 git 完全一致，只是选库）

| 命令 | 作用 |
|---|---|
| `agit <git…>` | 在代码仓库上跑 git（`status`/`commit`/`log`/`diff`/`push`…） |
| `agit -a <git…>` | 在 Agent Store 上跑同构 git（`add`/`commit`/`log`/`merge`/`push`/`pull`…） |

### agit 原生动词

| 命令 | 作用 |
|---|---|
| `agit init` | 初始化 Agent Store + 配对基建 |
| `agit clone <hub-url>/<name>.git` | 一条命令拉取团队 context（clone + 装驱动） |
| `agit -a new <主题> -e <证据> -m <结论> [--force]` | 手写一条带证据的 fact（落盘前做 subject 语义去重，`--force` 跳过） |
| `agit -a verify [--rerun]` | 重新校验所有证据（FRESH / STALE / MISSING…） |
| `agit -a why <主题>` | 出处链 + 当前状态 + 提交历史 |
| `agit -a merge <ref>` | 合并他人 context（冲突带证据裁决） |
| `agit -a resolve <主题> --take ours\|theirs` | 采纳冲突的某一侧 |
| `agit -a import [--from <rt>] [--summarize] [<session>]` | 从会话抽取 AgentState（见 §7） |
| `agit -a export [--to <rt>] [<文件>]` | 把 context 装回 runtime（见 §8） |
| `agit -a scan [--staged]` | 扫描密钥 |
| `agit -a validate` | 按 schema 校验 AgentState |
| `agit -a portable` | 导出 PortableState（跨机器/团队复用的引用） |
| `agit workspace [log]` | 看 Agent↔Environment 配对 |
| `agit adapter` | 列出 runtime adapter |

`agit-state`（demo 里附带的小脚本）随时告诉你「现在两个库 + 配对是什么样」。

## 6. 证据与 tier

每条 fact 至少带一条证据。证据的类型决定怎么校验、以及它的 **tier**（可逆性分层）：

| 证据写法 | 校验方式 | tier |
|---|---|---|
| `file:路径:行号[-行号]` | 重读那几行、重算摘要、和记录比对 | `reversible` |
| `cmd:命令` | 重跑命令（**默认不跑**，见下） | `compensable` |
| `doc:引用@YYYY-MM-DD` | 采集超过 365 天判定陈旧 | `reversible` |
| `human:谁@YYYY-MM-DD` | 人做的决策，不随代码失效 | `irreversible` |

`agit -a verify` 的状态：

| 状态 | 含义 |
|---|---|
| `FRESH` | 源头仍与采集时一致 |
| `RECHECK` | `cmd:` 证据，需 `--rerun` 才重跑 |
| `UNVERIFIABLE` | 人工决策，或没记摘要 |
| `STALE` | 源头变了，或文档超期 |
| `MISSING` | 源头不存在了 |

一条 fact 的状态取其证据里最强的那个。**只要有 STALE / MISSING，`verify` 退出码非零**——
可以直接挂 CI。

> **为什么 `cmd:` 默认不重跑**：一条从别人分支合并进来的 fact 可能带任意 shell 命令，
> `pull` 下来跑一次 `verify` 就等于执行陌生人的代码。必须显式 `agit -a verify --rerun`。

### subject 语义去重（落盘前）

冲突检测是句法的（比文件路径），所以「语义相同、subject 不同」会漏判——两条说同一件事的
结论换个名字就各存各的。为堵这个洞，`agit -a new` 在**落盘前**判断新结论是不是已有某条的同义：

```text
$ agit -a new api/user/identifier -e file:models/user.ts:4 -m '用户唯一标识就是 user_id 字段。'
疑似重复：这条结论和已有的 `api/user/id-field-name` 说的像是同一件事（claude 判定）。
  想补充/修正它 → 直接编辑 state/facts/api/user/id-field-name.md
  确实是另一条 → 加 --force 新建。
```

- **有 LLM 后端**（默认本机 `claude`）：让它从**已有 subject 里选**一个同义的（只能选、不能编）。
- **没有 LLM**：退回确定性启发式（同一父路径 + 正文词重叠），低置信提示。
- 这一步只出**建议**，`--force` 可覆盖；它**不进 merge 的确定性裁决路径**。

**换 LLM 后端（给 Codex 留的口子）：**

```sh
export AGIT_LLM=claude              # 默认，本机 claude -p
export AGIT_LLM_CMD="codex exec -"  # 任意从 stdin 读 prompt 的 CLI，立刻可用
```

同一个后端也被 `agit -a import --summarize` 复用。

## 7. 从会话抽取 context（import）

不想手写 fact，可以从一次 agent 会话里自动抽取。Claude Code 已实现，Codex 留了接口桩。

```bash
# 抽取当前项目最新的 Claude Code 会话（自动定位 ~/.claude/projects/<slug>/）
agit -a import

# 或指定会话文件
agit -a import path/to/session.jsonl

# 加 --summarize：额外调本机 claude 把「证据池」归纳成结论
agit -a import --summarize
```

抽取分两层：

1. **确定性层**（不调模型）：
   - **目标** ← 用户 prompt
   - **证据候选池** ← agent 的 Read（`file:` 证据）和 Bash（`cmd:` 证据）
   - `file:` 证据**当场对齐当前代码基线**重算摘要；`cmd:` 证据**只记不跑**
   - **产物** ← Write/Edit

2. **语义层**（`--summarize`，调本机 claude）：把证据池归纳成带出处的 fact。
   > **安全约束**：模型**只能引用证据池里 agent 真看过的文件、真跑过的命令**——
   > 编造出处在构造上不可能。池外的 locator 一律丢弃。

抽完审阅一下，然后提交：

```bash
agit -a add -A && agit -a commit -m '导入会话 context'
```

## 8. 把 context 装回 Claude Code（复用）

这是「一条命令在自己 runtime 复用」的落地。Claude Code 每个会话自动加载项目根的
`CLAUDE.md`，所以复用就是把 AgentState 写进 `CLAUDE.md` 的受管区块：

```bash
agit -a export --to claude-code
```

它在 `CLAUDE.md` 里写入一段 `<!-- agit:begin --> … <!-- agit:end -->` 受管区块：

```markdown
<!-- agit:begin —— 由 agit 管理，勿手改 -->
# 继承的 Agent Context（由 agit 从既往会话提炼）

## 目标
- 查清支付服务的用户模型与延迟成因

## 已知事实（带证据）
- **api/user/id-field-name** — 用户标识字段叫 user_id，不是 uid。
  - 依据 `file:models/user.ts:4 #a937b4a5`
...
<!-- agit:end -->
```

- **下一个 Claude Code 会话一开工就带着这份 context。**
- **幂等**：再次导出只替换受管区块，你手写在 `CLAUDE.md` 里的其它内容原样保留。
- 想喂给一次性会话：`claude -p "$(cat CLAUDE.md)"`。
- 写到别处：`agit -a export --to claude-code 某文件.md`。

> Codex 的 import 方向留了接口桩，拿到样本即可实现，上层命令不变。

## 9. 多人协作：合并与裁决

Agent Store 是普通 git 仓库，所以协作直接复用 git：

```bash
agit -a fetch                       # 看别人的 context
agit -a merge alice/context         # 合入完整更新
agit -a cherry-pick <revision>      # 只采一项更新
agit -a checkout -b 我的想法 <ref>   # 从已有 context 分叉
agit -a diff <ref-a> <ref-b>        # 比较两个 context
```

合并冲突时，agit 的 merge driver 会**在合并那一刻重新校验双方证据**，给出确定性建议：

```
<<<<<<< ours
用户标识字段叫 user_id。
=======
用户标识字段叫 uid。
>>>>>>> theirs

# ─────────────── agit 证据裁决 ───────────────
# ours   [FRESH] file:models/user.ts:4 —— 活代码
# theirs [STALE] doc:api-v1.md@2024-03-11 —— 两年前的文档
#
# 建议采纳: ours  （依据合并时的证据状态，非模型判断）
#   agit -a resolve api/user/id-field-name --take ours
```

采纳：

```bash
agit -a resolve api/user/id-field-name --take ours
agit -a add -A && agit -a commit -m 'merge：采纳 user_id'
```

> **裁决是确定性的**：同样的输入跑一万次结果一样，模型不进裁决路径。
> 双方证据强度相同时，它**拒绝猜**，把两条都摆给你判断。

## 10. AgentGitHub Hub

团队级的托管 + 浏览。详见 `docs/hub.md`。

### 启动（部署在一台机子上）

```bash
agit-hub add payments-api          # 托管一个 agent（建 bare 仓库）
agit-hub serve --port 8177         # 启动；默认数据目录 ~/.agit-hub
```

打开 `http://<那台机器>:8177/`：首页列出所有 agent（目标 + fact 数），点进去看某个 agent 的
目标、带出处的 fact、进度、历史，可搜索。

### 发布 context（Alice）

```bash
agit -a remote add origin http://<机器>:8177/payments-api.git
agit -a push -u origin main        # pre-push 先扫密钥，再走 git smart-http
```

### 消费 context（同事）

三种方式：

```bash
# A. 一条命令把整个 Agent Store 拉下来，对自己的代码基线复验
agit clone http://<机器>:8177/payments-api.git
agit -a verify

# B. 直接把「Claude Code 就绪」的 context 塞进 CLAUDE.md
curl -s http://<机器>:8177/agent/payments-api/claude.md >> CLAUDE.md

# C. agent 走 JSON
curl -s http://<机器>:8177/api/agent/payments-api
```

### Hub 端点一览

| 路径 | 内容 |
|---|---|
| `GET /` | 首页 |
| `GET /agent/<name>` | Context 页；`?q=` 搜 fact |
| `GET /agent/<name>/claude.md` | Claude Code 就绪的 context |
| `GET /api/agents` · `/api/agent/<name>` | JSON |
| `/<name>.git/...` | git smart-http（push/pull/clone） |

> **Hub 不重算新鲜度**：它没有你的代码，所以只做只读展示，新鲜度由消费者
> `agit -a verify` 对自己的基线判定。这是有意的。

## 11. 安全

- **`agit -a verify` 默认不执行 `cmd:` 证据**（`--rerun` 才跑）——防止 pull 别人 context 时
  执行陌生代码。
- **抽取时模型只能引用证据池内的东西**——编造出处在构造上不可能。
- **密钥三道防线**（因为 context 会被 push / pull）：
  1. 采集时拒绝：`.env`、`*.pem`、`id_rsa`、被 `.gitignore` 命中的路径，只记 locator 不抄内容
  2. 落盘前扫描：`agit -a new` 会扫正文和证据快照
  3. commit / push hook：`agit -a commit` / `agit -a push` 自动扫，`agit -a validate` 也扫
- **Hub 不保存 secret**：靠发布侧的 pre-push hook 拦。

## 12. 常见问题

**Q：`agit` 会替代 `git` 吗？**
不会。两者并存。`agit <git命令>` 只是帮你选对库，`agit add` 只暂存代码、`agit -a add`
只暂存 context，互不干扰。

**Q：`agit commit -a` 为什么提交的是代码不是 context？**
因为 `-a` 在子命令**之后**，是 git 的参数。要提交 context 用 `agit -a commit`（`-a` 在前）。

**Q：clone 别人的项目后 `agit -a` 报「未安装 merge driver」？**
跑一次 `agit init`。驱动配置不跟着 clone 走（git 安全设计）。

**Q：`agit -a verify` 全是 STALE？**
说明你的代码和 context 采集时的基线不一样了。这是**正确**的提示——那些结论需要复核。
更新结论：`agit -a new` 重新采集，或编辑 fact 后 `agit -a commit`。

**Q：Codex 能用吗？**
接口留了桩（`agit adapter` 能看到）。拿到 Codex 会话样本即可实现，上层命令不变。

**Q：`import ctx` 和手写 fact 有什么关系？**
`import` 是从会话自动抽取（省事但需审阅），`agit -a new` 是手写（精确）。两者产出同样的
fact 文件，可混用。

## 13. 完整工作流示例

一个「Alice 查清问题 → 发布 → 小林复用」的端到端例子：

```bash
# ── Alice（旧金山）──────────────────────────────
cd payments && agit init

# 让 agent 干活后，从会话抽取并归纳
agit -a import --summarize
agit -a verify                        # 确认证据都新鲜
agit -a add -A && agit -a commit -m '查清用户模型与延迟'

# 发布到团队 Hub
agit -a remote add origin http://hub:8177/payments-api.git
agit -a push -u origin main

# ── 小林（北京，第二天）──────────────────────────
cd payments                            # 他自己的代码 checkout
agit clone http://hub:8177/payments-api.git
agit -a verify                         # 对他的代码基线复验，看哪些还成立
agit -a why latency/order-service/n-plus-1   # 看 Alice 结论的出处

# 让他的 Claude Code 一开工就带着这份 context
agit -a export --to claude-code        # 写进 CLAUDE.md
# 现在启动 claude，新会话已经知道目标和已知事实

# 小林补充了新结论，推回去
agit -a new refund/flow -e file:services/refund.ts:8 -m '退款走三个服务。'
agit -a add -A && agit -a commit -m '补充退款流程'
agit -a push
```
