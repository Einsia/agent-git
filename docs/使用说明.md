# agit 使用说明

**给 AI 编程 agent 学到的东西做版本控制。**

代码通过 Git 和团队共享，但 agent 的工作没有。它弄清楚的一切 —— 读过哪些文件、排除了哪些死路、
某处改动背后的推理、下一步打算做什么 —— 都困在你机器上一个私有 session 里，然后随之消失。接手同一个
任务的同事，只能从零开始。

agit 把这个 session 当代码一样做版本控制。它快照 agent 本来就写到磁盘上的原始转录，让你 push/pull
同步；当两个人的 agent 出现分叉，它让 agent 自己去读对方的工作、完成合并，只把真正的冲突留给你裁决。
不用写摘要，不用维护 schema —— 只有 session 本身，在版本控制之下。

支持 **Claude Code** 和 **Codex**，两者完全对等。

---

## 工作原理

一个 **agent** 就是一段记忆：一个装着 session 转录的小型 git 仓库，以**它知道什么**命名
（`frontend`、`payments-api`），而不是以人或文件夹命名。一个 agent 可以在多个代码仓库里工作，一个
仓库也可以容纳多个 agent。

```
~/.agit/agents/agt_0190…/         agent —— 一个 git 仓库，以永不移动的 aid 标识
├── agent.toml                    它的身份，已提交，所以身份跟着历史一起走
└── sessions/claude-code/…        原始 session dump

your-project/                     你的代码，原封不动
├── .agit.toml                    已提交:这个仓库用哪些 agent(clone 读的就是它)
└── src/…
```

三条命令覆盖了大部分场景:

- `agit <git 命令>` 透明地对你的**代码**仓库跑 git。
- `agit a <git 命令>` 对解析到的那个 **agent** 的 store 跑 git。store 就是普通 git 仓库,所以
  `push` / `pull` / `clone` / `log` 都能用。
- `agit a merge <agent>` 把两个 agent 各自最新的 session 以只读方式复活,让它们通过阅读代码彼此对齐,
  产出一个**可 resume 的合并后 session** —— 而不是写进文件的摘要。

## 安装

```bash
npm install -g @einsia/agentgit
```

这会把 `agit`(以及 `agit-hub` 服务器)装到 PATH 上。或者从源码构建:

```bash
git clone https://github.com/Einsia/agent-git && cd agent-git
./build.sh --release
cp target/release/agit ~/.local/bin/
```

## 五分钟上手

在仓库里建立一个 agent:

```bash
cd your-repo
agit init --agent frontend      # 铸一个 agent 并绑定到这个仓库
```

照常工作,然后捕获并分享 agent 学到的东西:

```bash
agit start                      # 启动一个已带着该 agent 最新上下文的 session
agit snap                       # 把本项目的 session 收进 agent 的 store
agit a commit -m "auth flow"    # 它是 git 仓库 —— 提交这段记忆
agit a remote add origin https://hub.example.com/frontend.git
agit a push -u origin HEAD      # 推送,并把 remote 记进 .agit.toml 供团队使用
```

同事在刚 clone 下来的代码仓库里,一条命令就拿到同样的记忆:

```bash
agit a clone frontend           # .agit.toml 已写明用哪个 agent、在哪;这会把它 clone 下来
agit start                      # 从 agent 上次停下的地方继续
```

当两个人的工作出现分叉,对齐两个 agent:

```bash
agit a merge frontend
# 两个 agent 都复活,读彼此的 session 和代码,能解决的都解决掉。
# 只有真冲突才停下来问你。产物是一个可 resume 的 session:
#   claude --resume <id>   或   codex exec resume <id>
```

凡是 git 动词在 store 上意义相同的,`agit a` 就沿用 git 的名字并加上 agent 相关的行为:
`agit a clone`(按身份)、`agit a init`(铸一个)、`agit a switch`(选活跃 agent)、
`agit a push`/`pull`/`fetch`/`merge`。其余是固定的管理命令(`list`、`info`、`rename`、`rebind`)。
`a` 后面其它任何词都原样交给 git,所以 `agit a log`、`agit a diff` 的行为和你预期一致。agit 从不取代 `git`。

## 值得了解的几点

**身份。** agent 由 `aid`(`agt_<uuid>`)标识,铸造一次并提交在它自己的 store 里。不是名字
(名字是可变标签,会撞车),也不是 URL(URL 只是定位符 —— 你可以在 agent 还没有 remote 的时候就
创建它)。因为 `.agit.toml` 记的是 aid,即使有人重建了一个同名的 remote,也不会把你悄悄绑到另一个
agent 上。

**选中哪个 agent。** 任何 agent 相关的命令,都按固定顺序解析,从不静默兜底:

1. `--agent <name|aid>` —— 逐条命令生效
2. `$AGIT_AGENT` —— 逐 shell(这也是同时跑两个 agent 的一种方式)
3. **活跃指针** —— 每个 worktree 一份,本地、不入库(可以放心删,删掉就退回下一档)
4. `.agit.toml` 的 `[defaults]` —— 新克隆默认激活的那个
5. 一条可执行的报错

**同时跑两个 agent。** 选择是逐条命令生效的,所以你可以在同一个仓库里并排跑两个:

```bash
agit start --agent frontend     # 终端 1
agit start --agent api          # 终端 2
```

agit 按 `agit start` 写下的 launch record 归属每次捕获的 session,所以即便两个运行时都 dump 到同一个
文件夹,也不会混。

**运行时对等。** Claude Code 和 Codex 是平等的。读取 session 的命令用 `--from` 指定其一,否则用
现存的唯一一个,否则询问你。

**全自动捕获。** `agit watch --daemon` 在后台自动 snap、自动 convert,你不用记着去跑 `snap`
(`--stop` / `--status` 管理)。

**换驱动合并的模型。** merge 用到的模型统一走一个可插拔后端:`AGIT_LLM=claude`(默认,本机
`claude -p`)、`AGIT_LLM=codex`(本机 `codex exec`),或 `AGIT_LLM_CMD="<从 stdin 读 prompt 的
任意命令>"`(例如接 ollama)。

## 合并是怎么发生的

session 是不同 uuid 的文件,所以 `git pull` 拿到对方的 session 时不会产生文本冲突 —— 它们只是并排
落进 `sessions/`。真正的“合并”发生在语义层,由 `agit a merge` 完成。**它的操作数永远是另一段记忆**
(一个 agent 名,或者你 store 里的一个 ref),从不是代码分支。

合并模式由**身份**决定,而不是靠 git 历史去猜。因为 `agent.toml` 提交在 store 内部,任何目标的
aid 都读得到:

- **aid 相同**(是我这个 agent 的另一份副本,比如同事 push 上来的)→ 先对话对齐,再把两边历史一起
  merge,重新变回一段记忆。
- **aid 不同**(是另一个 agent)→ 只对话,两份记忆都完整保留。

过程是这样的:两条真实 session 同时以只读方式复活,各自在自己分支的 git worktree 里,把“自共同祖先
以来的 diff”作为事实依据传入。两个 agent 通过阅读代码厘清差异,能解决的都解决,只把真冲突拿出来一条条
交给你裁决。产物是一个**可 resume 的 session**(不是写进 `CLAUDE.md` 的摘要):merge 结束会打印
resume 命令(`claude --resume <id>` / `codex exec resume <id>`),你 resume 它,就带着双方已对齐的
上下文继续。

因为对话由真实的运行时 session 驱动,merge 需要相应的运行时 CLI 可用。`--from` 指定复活哪一侧的
运行时,`--both` 把合并后的 session 写到两条分支上,`--quick` 跳过上下文交接。

## 命令速查

### 顶层命令(作用在代码仓库这一侧,或管理 session)

| 命令 | 作用 |
|---|---|
| `agit init [--agent <name>]` | 让仓库能用 agent:装 gitignore、绑定一个 agent、装密钥 hook |
| `agit start [--agent <n>] [--as <rt>]` | 在**这里**启动一个已带着该 agent 最新上下文的 session,并写下 launch record |
| `agit snap [--from <rt>] [--no-harness]` | 把本项目的 session dump + harness(MCP/skills/config,密钥已脱敏)镜像进**所属 agent** 的 store |
| `agit watch [--daemon]` | 全自动:盯着两个运行时,自动 snap + 自动双向 convert(`--stop` / `--status` 管理) |
| `agit convert <src> --to <rt> [--write]` | 把一份 session 转成另一个运行时能 resume 的会话(默认 dry-run) |
| `agit resume <src> [--as <rt>] [--exec]` | 把一份 session 装回运行时并继续 |
| `agit workspace [log]` | 查看 Agent↔Environment 配对 |
| `agit workspace restore [N]` | 把两个仓库一起退回某条配对记录的联合状态 |
| `agit graph` | Workspace-State 时间线 + 关系边 |
| `agit harness [apply]` | 查看,或应用,已捕获的 harness(MCP/skills/config) |
| `agit adapter` | 列出运行时 adapter(claude-code、codex) |
| `agit <git 命令>` | 透明地对代码仓库跑 git(如 `agit clone`、`agit commit -a`) |

### `agit a <verb>` —— agent 管理(封闭集合)

| 命令 | 作用 |
|---|---|
| `agit a list` | 本机有哪些 agent |
| `agit a init <name>` | 铸一个 agent(不需要任何 remote —— 身份先于 URL) |
| `agit a switch <name>` | 设定这个 worktree 的默认 agent(是默认,不是锁) |
| `agit a clone <name\|url>` | 取回一个本机还没有的 agent:裸名字去已提交的 `.agit.toml` 里查 remote,或直接给 URL |
| `agit a info <name>` | 查看某个 agent 的 aid / store / remote |
| `agit a rename <old> <new>` | 只改元数据(store 以 aid 为键,目录不动) |
| `agit a rebind <name> --remote <url>` | 覆盖身份完整性检查;`--new-id` 给 fork 一个新身份 |
| `agit a merge <target>` | 对话式合并两段记忆(见上) |
| 其它 `agit a <git 命令>` | 原样的 git:`agit a commit`、`agit a push`、`agit a pull`、`agit a log`、`agit a diff`… |

## 协作:Hub

`agit-hub` 是一个自带的服务器,把 agent 作为裸 git 仓库托管,提供同步和一个 web UI(编译进二进制的
React 应用)。它给每个 session 一条可浏览的事件时间线、来源信息、永久链接和 revision diff,并提供两个
跨 agent 的视图:组织总览(`/orgs/<name>`:成员及各自你可见的 agent 与其涉及的代码仓库)和代码仓库索引
(`/repos`:把各 agent 工作过的代码仓库按环境归组、列出关联的每个 agent)。

权限**以 agent 为单位**:每个 agent 有一个 owner、一个可见性(默认私有)、以及带 read / write /
admin 的成员。人用密码登录(argon2id + cookie session);脚本和 git 用 token,token 可以限定到单个
agent、设过期、可吊销。每个请求 —— 包括 git smart-http —— 都走同一个授权判断。每次 push 都会在服务端
扫描密钥,所以泄漏的凭据进不了共享仓库。

发布和消费都走普通的客户端命令:一端 `agit a remote add origin <url>` + `agit a push` 记下 remote 并推送,另一端
`agit a clone <name>` 从 `.agit.toml` 里读出 remote 并 clone。要自己跑一个 hub,见
[deploying-the-hub.md](deploying-the-hub.md)(`deploy/` 下有 Docker / Compose 配置);权限模型和
API 的细节见 [hub.md](hub.md)。

## 安全

- **session 里可能有密钥** —— agent 读过的 `.env`、打印出来的 token。每个 store 在每次 commit 和
  push 前都会被扫描,到了 hub 还会在服务端再扫一遍,所以密钥不会跟着走。
- **来源(provenance)** —— 每次捕获的 session 都用本机 ed25519 密钥签名,签名里记录你的 git 身份
  (`user.email`)作为归属句柄;身份未设置时 `agit snap` 会拒绝快照,而不是把 session 归于无人(创建
  agent 不需要身份,只有快照需要)。把设备密钥用 `agit identity register` 注册到 hub 账号后,
  `agit provenance verify` 可把校验升级为 **VERIFIED AS 某人**。
- **`.agit.toml` 是攻击者可控的输入**(是别人写的)。agit 在 clone 一个 remote 之前,会拿它和一份
  传输协议白名单比对,因为 `git clone 'ext::<cmd>'` 会执行 `<cmd>`,而 `--` 挡不住。
- **merge 用到模型**,所以它按设计就是非确定性的 —— 这是换来一个无需维护 schema 的真正语义合并所付的
  代价。它会合并所有能安全合并的东西,只在真冲突处停下。

每一条的细节和缓解方向,见 [风险分析.md](风险分析.md)。
