## 1. 它解决什么问题

代码通过 Git 对团队公开了,但 agent 的工作困在私人 session 里 —— 它读过什么、做过什么判断、
下一步是什么,别人看不到、没法复用。

agit 直接管理 agent 的**原始会话**:claude-code 与 codex 本来就把整个会话 dump 到磁盘,agit 把这坨
版本化、push/pull 同步;合并时让**另一个 agent 读懂对面的会话、自行合并**,只有真冲突才问你。

> 之前有过一版"手写 fact + 证据 + 确定性合并"的模型,已废弃。为什么、以及新旧取舍见
> [`风险分析.md`](风险分析.md)。

## 2. 核心模型:agent 是一段记忆

**agent 不是人,也不是项目里的一个文件夹。** 它是一段记忆:一个装满转录的 git 仓库,以**它知道什么**
命名(`frontend`、`payments-api`)。

```
~/.agit/agents/agt_0190f3a1-…/     ← agent:一个独立 git 仓库,以身份为键
├── agent.toml                     ←   aid,已提交 —— 所以身份跟着历史一起走
└── sessions/claude-code/
    ├── <id>.jsonl                       完整转录
    └── <id>/subagents,tool-results

你的项目/                          ← Environment:你的代码,agit 原封不动
├── src/auth.js
├── .agit.toml                     ← **已提交**:这个仓库用哪些 agent。协作靠的就是它。
└── .agit/                         ← 本地状态(workspace 配对、watcher 的 pid/log),自动 gitignore
```

三条推论,构成整个模型:

- **多对多。** 一个 agent 在多个仓库里工作,一个仓库容纳多个 agent。所以 store 不能放在代码仓库里
  —— 那正是"让一个 agent 把上下文带进另一个仓库"过去做不到的原因。
- **身份是 `aid`**(`agt_<uuid>`),铸造一次,提交在 store 自己的 `agent.toml` 里。
  **不是名字**(名字是标签,会撞车);**不是 URL**(URL 只是定位符 —— 你在任何 remote 存在之前就已经
  铸造了这个 agent;换 hub 会重写所有 URL;镜像会让一个 agent 有两个 URL)。
- **store 以 aid 为键** ⇒ 改名和发布都只是元数据改动,**目录永不移动**,正在运行的 watcher 永不被孤立。

### 两个 scope

```
agit <git 命令>       →  透明作用在你的代码仓库(Environment)
agit a <git 命令>     →  同构作用在**解析到的那个 agent** 的 store
```

`a` 是子命令,不能对调位置,所以旧版最大的坑消失了:`agit a commit`(agent 库)vs
`agit commit -a`(代码库,`-a` 是 git 的 stage-all)。旧的 `agit -a <args>` 仍作为**弃用别名**保留。

store 就是 git 仓库,所以 `push` / `pull` / `clone` / `fetch` / `log` / `diff` 全部透传白拿。

### 两个运行时是对等的

claude-code 与 codex,永远按字母序一起出现。**没有默认值**:读取会话的命令用 `--from` 指定,
否则用现存的唯一一个,否则询问。("`snap` 悄悄等于 claude" 是一个 framing bug,已修。)

## 3. 安装

```bash
./build.sh --release              # 为什么不是 cargo build:见根目录 README
cp target/release/agit ~/.local/bin/

cd 你的项目
agit init                         # 问你 agent 叫什么,然后绑定在这个仓库
```

`agit init` 做三件事:把 `.agit/` 加进 `.gitignore`;确保这个仓库有一个能解析到的 agent
(没有就铸一个,会问你名字 —— `--agent <name>` 可跳过,非交互时用目录名);把密钥 hook 装到
store 上。它是幂等的。

## 4. 五分钟上手

```bash
cd 你的项目
agit a new frontend                             # 铸一个 agent(此时还不需要任何 remote)
git add .agit.toml && git commit -m 'agit: 绑定 frontend'   # 提交绑定,队友克隆即可见

agit start                                      # 启动一个已经带着该 agent 最新上下文的会话
agit snap                                       # 把本项目的 session dump + harness 收进 store
agit a add -A && agit a commit -m '记录会话'
agit a push                                     # 发布(push 前扫密钥)

# 同事那边(刚 clone 完代码仓库)
agit a track frontend                           # .agit.toml 已说明用哪些 agent、在哪 —— 这就把记忆拉下来
agit a merge frontend                           # 两个 agent 对话对齐,真冲突才问你
```

## 5. 命令详解

### 顶层动词(作用于代码仓库这一侧)

| 命令 | 作用 |
|---|---|
| `agit init [--agent <name>]` | 让这个仓库能用 agent:gitignore + 绑定一个 agent + 装密钥 hook |
| `agit start [--agent <n>] [--as <rt>]` | 在**这里**启动一个已带着该 agent 最新上下文的会话(不管它上次在哪个仓库);同时写下 launch record 并确保 watcher 在跑 |
| `agit snap [--from <rt>] [--no-harness]` | 把本项目的 session dump + harness(MCP/skills/config,密钥已脱敏)镜像进**所属 agent** 的 store |
| `agit watch [--daemon]` | 全自动:盯着两个运行时,自动 snap + 自动双向 convert(`--stop` / `--status` 管理) |
| `agit convert <src> --to <rt> [--write]` | 把一份 session 转成另一个 runtime 能 resume 的会话(默认 dry-run) |
| `agit resume <src> [--as <rt>] [--exec]` | 把一份 session 装回 runtime 并继续 |
| `agit workspace [log]` | 看 Agent↔Environment 配对 |
| `agit workspace restore [N]` | 把两个库一起退回某条配对记录的联合状态 |
| `agit graph` | Workspace-State 时间线 + 关系边 |
| `agit harness [show\|apply]` | 看/应用已捕获的 harness |
| `agit adapter` | 列出 runtime adapter(claude-code、codex,均已实现) |

### `agit a <verb>` —— agent 管理,一个**封闭集合**

| 命令 | 作用 |
|---|---|
| `agit a list` | 本机有哪些 agent(`●` 是本 worktree 当前默认) |
| `agit a new <name>` | 铸一个 agent。**没有 remote 也能用** —— 身份先于任何 URL |
| `agit a use <name>` | 设定**我在这个 worktree** 的默认。是默认,不是锁 |
| `agit a track <name\|url>` | 取回一个本机还没有的 agent。裸名字去**已提交的** `.agit.toml` 里查 —— 这就是新克隆拿到团队 agent 的路径。默认顺带激活(`--no-use` 可关) |
| `agit a info <name>` | 看某个 agent 的 aid / store / remote |
| `agit a rename <old> <new>` | 只改元数据(store 以 aid 为键,目录不动) |
| `agit a merge <target>` | 对话式合并两段记忆(§7) |
| `agit a import` | 收编旧版遗留的 `.agit/agent`(§10) |
| `agit a publish` / `a rebind` | 尚未实现(设计见 design of record §5) |

> 集合是封闭且刻意避开 git 命名空间的:`track` 而不是 `add`(`git add` 太常用,不能被遮蔽),
> `info` 而不是 `show`(`git show` 是真动词)。**`a` 后面不在这个集合里的词一律交给 git** ——
> 所以 `agit a log`、`agit a add -A`、`agit a commit`、`agit a show` 全是原样的 git。

### 同时跑两个 agent

`use` 设的是默认,不是锁。选择是逐次的:

```bash
# 终端 1                          # 终端 2,同一个仓库,同一时刻
agit start --agent frontend       agit start --agent api
```

`--agent` **不会**翻转默认值。捕获时按 `agit start` 写下的 **launch record** 归属,而不是按当前
默认指针 —— 两个 agent 共用同一个 dump 目录,按指针归属会**静默归错档**,推给错误的团队。

## 6. 选中哪个 agent:解析顺序

任何 agent 相关的命令,都按这个顺序:

1. `--agent <name|aid>` —— 逐次
2. `$AGIT_AGENT` —— 逐 shell(这是同时跑两个 agent 的方式)
3. **活跃指针** —— 每个 worktree 一份,本地、不入库
4. `.agit.toml` 的 `[defaults] agent` —— **新克隆**激活的那个
5. 可执行的报错。**永不静默兜底。**

> 活跃指针可以放心删:删掉就退回 `[defaults]`。这是它与旧版 `.agit/store` 指针文件的根本区别 ——
> 一个 gitignore 的本地文件是正当的,**当且仅当它的缺失可以完全从已提交的状态恢复**。
> `.agit/store` 存的是一个绝对路径,又是唯一的解析依据:删了它,没有任何东西还能说出 store 去哪了。

## 7. merge:合并是怎么发生的

会话是不同 uuid 的文件,所以 `git pull/fetch` 拿到对面的会话时**不会文本冲突**,它们只是并排落进
`sessions/`。真正的"合并"是语义层的,由 `merge` 做。**它的操作数永远是另一段记忆,不是代码分支。**

```bash
agit a merge <X>
  ├─ X 是已知的 agent 名?  → 那个 agent 的 store
  ├─ X 是我 store 里的 ref? → 那个 ref
  ├─ 两者都是             → 让你选(脚本用 --agent X / --ref X)
  └─ 都不是               → 报错 + 建议
```

### 模式由**身份**决定,而不是由 git 历史猜

`agent.toml` 提交在 store **内部**,所以任何目标的 aid 都读得到:

| | aid 相同(我的 agent 的另一份副本) | aid 不同(另一个 agent) |
|---|---|---|
| 例子 | `origin` —— 队友 push 上来的 | `frontend` |
| 结果 | 对话 → **融合**:连历史一起 git merge,重新变回一段记忆 | 只对话 → **两份记忆都完整保留** |

```console
$ agit a merge origin
origin is this agent (agt_01J…) — reconciling, then merging the histories.

$ agit a merge frontend
frontend is a different agent (agt_02X…) — reconciling by dialogue; histories stay separate.
```

跨 agent 的产物 = 一个**可 resume 的合并后会话** + 一份归档的对话转录;代码你自己接着合 ——
这正是「同步一下上下文,然后接着合并」。

### 过程

1. 把两条**真实会话同时复活**(只读),各自运行在其所属分支的 worktree 中,并把「自公共祖先以来的
   diff」作为 ground truth 传入。
2. 两个 agent 通过**阅读代码**厘清差异。双方都带着完整上下文,所以交叉冲突会浮现 —— 一次性摘要
   合并会把它遗漏。
3. 产物是一个可 resume 的会话(**不是**写进 `CLAUDE.md` 的摘要):resume 它即可带着双方已对齐的
   上下文继续。真冲突谈不拢时会停下来请你裁决,裁决结果写进该合并会话。

> **有模型的那一步是非确定性的**(交给模型),同样输入两次结果可能不同。这是取舍。
> 它会在什么 case 上出问题(悄悄合错、不可复现、prompt 注入……)见
> [`风险分析.md`](风险分析.md) —— 用之前值得读一遍。

## 8. 换 LLM 后端

merge 用到的模型统一走 `src/llm.rs`,可插拔:

```sh
export AGIT_LLM=claude               # 默认,本机 claude -p
export AGIT_LLM=codex                # 本机 codex exec
export AGIT_LLM_CMD="<从 stdin 读 prompt 的任意命令>"   # 覆盖一切,如 ollama
```

merge 的两侧都是**真实 resume 的会话**,所以对应的 runtime CLI 必须在。没有可用 LLM 后端时,
合成那一步跳过:未决冲突列给你,而不是自动定夺。

**Codex 是对等 runtime**(不是桩):它的 session dump(`~/.codex/sessions/`,按 `cwd` 过滤本项目)
能 `agit snap --from codex`、`agit a merge <agent> --from codex`、`agit convert`。
解析在 `src/adapter/codex.rs`。

> ⚠️ **codex 的 install id 永远是 UUID。** 用专有名字做 thread id 时,`codex exec resume <name>`
> 会**静默地开一个全新空会话并 exit 0** —— 这正是 agit 存在的意义所要防的失败模式。详见
> design of record §9。

## 9. Hub

团队级托管 + 浏览,详见 [`hub.md`](hub.md)。要点:Hub 只做 **Registry + Sync + 只读渲染**,
**不合并**(合并在消费者本地 merge)。前端是内嵌的 React SPA(会话脊线 / provenance / permalink /
revision diff)。**权限以 agent 为单位**:每个 agent 有 owner、可见性(private/public)与成员
(read/write/admin);git 与脚本用 token,可绑定到单个 agent、设 TTL、可吊销。**默认私有**,
默认只监听回环。

## 10. 从旧版 agit 迁移

旧版的 store 在 `<你的仓库>/.agit/agent`,被焊死在一个代码仓库上。**它已经不再被解析** —— 那道焊缝
正是"让一个 agent 在另一个仓库里接着干"做不到的原因。收编它:

```bash
agit watch --stop     # 若有 watcher 在跑:store 从它脚下被移走会把它变成僵尸,captures 静默丢失
agit a import         # 铸身份 + 移动 store + 写 .agit.toml,什么都不会丢
```

`import` 会在 watcher 还活着时**拒绝执行**并告诉你先停掉它。任何 agent 相关的命令都会给出同一条
可执行的提示,所以你不管敲的是 `agit a log` 还是 `agit snap`,都会被告知该跑 `import`。

一并没了的还有:`agit init --store <path>` 与 `$AGIT_AGENT_DIR`。它们回答的都是"store 在**哪**",
而新模型只回答"这是**谁**"——每个 agent 本来就是分离的,`--store` 想买的东西已经白送。

## 11. 安全与已知局限

- **`agit a scan` / commit / push hook** 扫 session 里的密钥。dump 全会话 = agent cat 过的
  `.env`、连接串都在转录里,不扫就发出去。这道闸门在 **store 被创建时**就装上,无论它从哪扇门来
  (`a new`、`a track`、`a import`、`init`)—— 漏装钩子的 store 会**静默地**什么都不扫。
  jsonl 按 JSON 解析、只扫**字符串值**,并按字段名和 shape(uuid / sha / 时间戳 / 路径 / 内容寻址
  哈希)挡掉转录噪声,所以熵检测能开着;但**拦不住一般性敏感内容**(商业机密、PII)。
- **误报的逃生口**(没有它,人只会去用 `--no-verify` 把整道闸门关掉):
  行内加 `agit:allow-secret` 豁免该行;或在被扫根目录放 `.agit-allow-secrets`,
  每行一条字面子串、或 `re:` 开头的正则。
- **`.agit.toml` 里的 remote 是攻击者可控输入。** 它是**提交进版本库**的,所以 `agit a track frontend`
  clone 的是"写这个仓库的人"选的 URL。已验证(git 2.43):`git clone 'ext::<cmd>'` **会执行 `<cmd>`**,
  且 `--` 挡不住(`ext::` 是 scheme,不是 flag)。故 agit 用**传输协议白名单**拒绝而非净化:
  分类不了的 URL 就是担保不了的 URL,让人自己粘贴。
- **agent 的 remote 就是权限边界。** 一个 agent 一个仓库,谁能读它取决于谁能读那个仓库。
  分支**不是** ACL 边界 —— 这也是"一个 agent 一个仓库"的独立理由。
- **merge 非确定性**、依赖外部模型、把不可信会话喂给 LLM(prompt 注入面)。
- **丢了旧模型的"证据→代码"过期检测**。
- 以上每一条的细节和缓解方向,见 [`风险分析.md`](风险分析.md)。

## 12. 完整工作流示例

```bash
# ── Alice ────────────────────────────────
cd payments && agit init --agent payments-api
agit start                                     # 会话带着 payments-api 的最新上下文起来
agit snap                                      # 镜像她这次会话
agit a add -A && agit a commit -m '查清用户模型'
# Hub 管理员先发一个写 token:agit-hub token add alice --agent payments-api --write（token 只显示一次）
agit a remote add origin http://alice:<token>@hub:8177/payments-api.git
agit a push -u origin main
agit a track payments-api                      # 把 remote 记进 .agit.toml
git add .agit.toml && git commit -m 'agit: 绑定 payments-api' && git push

# ── 小林(第二天,刚 clone 完代码仓库)────────
cd payments
cat .agit.toml                                 # 这个仓库用哪些 agent,一目了然
agit a track payments-api                      # 按名字取回记忆(remote 就写在绑定里)
agit start                                     # 直接接着 Alice 的上下文干
agit snap && agit a add -A && agit a commit -m '我这次会话'

# ── 两个人分头做完,记忆分叉了 ───────────────
agit a fetch origin
agit a merge origin/main                       # 同一个 agent 的两份副本 ⇒ 对话之后连历史一起融合
```

```bash
# ── 跨端迁移(PRD #3):frontend agent 接着在 backend 仓库干 ──
cd ~/code/api
agit a track frontend && agit start            # 带着它在 ~/code/web 的最新会话进来
agit a log                                     # 一段记忆,两个环境
```
