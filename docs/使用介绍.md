# Agent git 使用介绍

agit 用 git 管理 AI coding agent 的 session。

代码在 git 里，产生代码的那次 session 不在。Claude Code 或 Codex 读取过哪些文件、排除了哪些方案、为何这样改动、下一步打算做什么，这些都保留在本地的一份转录里，同事拉取分支时看到的只是结果。agit 把这份 session 也放进 git：快照进一个仓库，用 push 与 pull 同步；当两个人的 session 出现分叉，由双方的 agent 读取彼此的工作、对照代码完成合并。它支持 Claude Code 与 Codex。

## 基本模型

每个 agent 是一个存放 session 转录的 git 仓库，位于 `~/.agit/agents/<aid>/`。你为它起一个说明用途的名字，例如 `frontend`、`payments-api`。它有一个固定身份 `aid`，记录在自身的 `agent.toml` 中。

代码仓库中会新增一个 `.agit.toml`，记录用到哪些 agent、以及从何处克隆，并提交进 git，同事克隆代码仓库时一并获得。一个 agent 可以用于多个仓库，一个仓库也可以有多个 agent。

agit 是在 git 之上加了一层，而不是取代它。`agit <git 命令>` 就是对你的代码仓库跑 git，原本敲的命令照常可用，`agit commit`、`agit push` 都一样。在 `agit` 后面加一个 `a`，同一条命令就改为作用在当前 agent 的 store 上；store 是一个普通 git 仓库，`agit a log`、`agit a diff`、`agit a commit` 也都照常可用。

## 安装

```bash
npm install -g @einsia/agentgit
```

安装的是 `agit` 客户端。agit-hub 服务端通过 Docker 或源码构建单独获取。

## 日常使用

先为仓库初始化一个 agent，启动后台守护进程，之后照常编写代码：

```bash
agit init --agent frontend   # 初始化该仓库，并创建第一个 agent
agit watch --daemon          # 从此在后台自动捕获
```

`agit watch --daemon` 在后台自动把新的 session 收入对应 agent 的 store，并转换成另一个 runtime 的格式，使 Claude Code 中的 session 可在 Codex 中 resume，反之亦然。用 `agit watch --status` 查看状态，`agit watch --stop` 停止。

不使用守护进程时，用 `agit snap` 手动执行一次。`agit start` 在当前目录启动一个已携带该 agent 最新上下文的 session。

## agent store 上的命令

store 上的命令沿用 git 的名字，agent 版本会在此之外多做一步：

| 命令 | 说明 |
|---|---|
| `agit a init <name>` | 在这个仓库里再加一个 agent（一个带独立身份的 store） |
| `agit a clone <name\|url>` | 按身份克隆一个 agent，裸名字在 `.agit.toml` 中查找 |
| `agit a switch <name>` | 指定当前 worktree 使用哪个 agent |
| `agit a push` | 推送，并把 store 的 origin 记入 `.agit.toml`，剥除凭据 |
| `agit a pull` | 只做 fast-forward，分叉时转交 `agit a merge` |
| `agit a fetch` | 拉取，并报告新到达的 session |
| `agit a merge <target>` | 以对话方式合并两段记忆 |

其余是 git 中没有对应物的管理命令：`agit a list` 列出本机的 agent，`agit a info <name>` 查看某个 agent 的信息，`agit a rename <old> <new>` 重命名，`agit a rebind` 修复身份、或用 `--new-id` 为一个 fork 赋予新身份。`a` 之后的其它任何词都交给 git。

## 跨 runtime

agit 支持 Claude Code 与 Codex。读取 session 的命令用 `--from` 选定其一，只安装了一个就用它，两个都在则询问。`agit adapter` 列出已识别的 runtime。

`agit convert <src> --to <runtime>` 把一份 session 转换成另一个 runtime 能 resume 的格式，`agit resume <src>` 把一份 session 载入 runtime 并继续。守护进程会自动完成这项转换。

## 合并

`agit a merge <target>` 让两边最新的 session 以只读方式复活，对照代码合并各自的工作，产出一个可 resume 的合并后 session，并附上需要你裁决的冲突。用 `claude --resume <id>` 或 `codex exec resume <id>` 继续。

如果目标是同一个 agent 的另一份，例如同事推送上来的，git 历史也会一并合并；如果是另一个 agent，只合并对话，两侧各自保留。合并借助模型完成，因此结果并不确定。

不想动用模型时，可以加 `--splice`。它不做对照，只把两边拼成一份新的 session：A 的完整记录，接上 B 从分叉点之后的部分，session id 归一之后安装为一个可 resume 的 session。resume 它，新的 agent 会在同一个上下文里同时读到两边，再自行决定怎么处理。这条路不经过模型，也不需要 runtime CLI，结果是确定的。

## 协作与 hub

分享只需为 store 配置一个 remote 并 push：

```bash
agit a remote add origin https://hub.example.com/frontend.git
agit a push -u origin HEAD
git add .agit.toml && git commit
```

`agit a push` 会把 remote 记入 `.agit.toml`。同事克隆代码仓库后，运行 `agit init` 就能一次取回绑定中声明的全部 agent；只取其中一个则用 `agit a clone frontend`。`agit harness apply` 把这个 agent 的 MCP server、skills 与配置一并带到本地。

想为团队提供一个带 web 界面和权限管理的服务端，可以自托管 **agit-hub**。它把 agent 作为裸 git 仓库托管，提供同步和一个 web UI，把每个 session 呈现为可浏览的事件时间线；权限按 agent 划分，每次 push 在服务端扫描密钥，操作留有审计。agit-hub 单独发布，通过 Docker 或源码构建，只负责托管与渲染，不运行 agent，也不执行合并。

## git shadow

`agit shadow install` 让 shell 中的 `git` 经过 `agit`，于是每条 git 命令都一并记录 agent 上下文。支持 bash、zsh、fish 与 PowerShell，`agit shadow uninstall` 撤销。

## 安全

session 中可能带有密钥，例如 agent 读取过的 `.env` 或打印出的 token，因此每个 store 在 commit 与 push 之前都会扫描，推送到 hub 后再扫描一次。`.agit.toml` 是他人写入的内容，agit 在克隆它声明的 remote 之前会先对照一份传输协议白名单。
