# Agent git 使用介绍

代码本身早已处于版本控制之下，产生这些代码的 session 却没有。Claude Code 或 Codex 在一次会话中读取过哪些文件、尝试并排除了哪些方案、某处改动的依据、以及原定的下一步，这些信息都留在本地的一份转录里。同事次日拉取该分支时，得到的只是结果，而非其中的推理过程。

agit 的做法是把这份 session 一并纳入 git 管理。它将 session 快照进一个仓库，通过 push 与 pull 同步；当两人的 session 出现分叉时，由 agent 各自读取对方的工作、对照代码进行对齐，只把真正冲突之处交给使用者裁决。agit 同时支持 Claude Code 与 Codex。

## 基本模型

一个 **agent** 就是一个存放 session 转录的 git 仓库，位于 `~/.agit/agents/<aid>/`（即 `$AGIT_HOME`）之下。它以“所掌握的内容”命名，例如 `frontend`、`payments-api`，而不是以人或目录命名。它的身份是一枚 `aid`（形如 `agt_<uuid>`），只在创建时铸造一次，提交在自身的 `agent.toml` 里；此后名称与 remote 都可以变，唯有 aid 始终不变。

代码仓库本身几乎不受影响，只多出一个纳入提交的 `.agit.toml`，用来声明该仓库使用了哪些 agent、以及从何处克隆。他人克隆代码仓库时读取的正是这个文件。agent 与代码仓库之间是多对多的：一个 agent 可以服务于多个仓库，一个仓库也可以关联多个 agent。

命令有两个 scope。`agit <git 命令>` 透明地作用于代码仓库；`agit a <git 命令>` 则作用于所解析出的那个 agent 的 store。store 本身就是一个普通 git 仓库，因此 `agit a log`、`agit a diff`、`agit a commit` 等命令都可以照常使用。

## 日常使用

多数情况下，只需启动一次后台守护进程，之后照常编写代码即可：

```bash
agit init --agent frontend   # 为该仓库铸造一个 agent 并完成绑定
agit watch --daemon          # 启动守护进程，此后无需干预
```

`agit watch --daemon` 会在后台持续运行，自动把新产生的 session 收入对应 agent 的 store，并将其转换成另一 runtime 的格式，使在 Claude Code 中记录的 session 也能在 Codex 中 resume，反之亦然。查看它的运行状态与已捕获的内容可用 `agit watch --status`，停止则用 `agit watch --stop`。

若不使用守护进程，`agit snap` 就是把上述捕获步骤手动执行一次。

如需在当前目录启动一个已携带该 agent 最新上下文的 session，使用 `agit start`，其中 `--agent` 指定 agent，`--as` 指定 runtime。

## agent store 上的命令

凡是 git 动词在 store 上语义一致的，agit 都沿用它的名称，agent 版本只是在此基础上多做一步必要的处理：

| 命令 | 说明 |
|---|---|
| `agit a init <name>` | 铸造一个新 agent，也就是一个带有独立身份的 store |
| `agit a clone <name\|url>` | 按身份克隆一个 agent；裸名称会在 `.agit.toml` 中查找 |
| `agit a switch <name>` | 指定当前 worktree 使用哪个 agent |
| `agit a push` | 推送，并把 store 的 origin 记入 `.agit.toml`，其中的凭据会被剥除 |
| `agit a pull` | 只做 fast-forward；一旦分叉就转交 `agit a merge` |
| `agit a fetch` | 拉取，并报告有哪些新 session 到达 |
| `agit a merge <target>` | 以对话的方式合并两段记忆 |

其余几个是 agit 特有、git 里并没有对应物的管理命令：`agit a list` 列出本机的 agent，`agit a info <name>` 查看某个 agent 的 aid、store 与 remote，`agit a rename <old> <new>` 只改名称而不动 aid，`agit a rebind` 用于覆盖身份完整性检查，或通过 `--new-id` 为一个 fork 赋予新的身份。`a` 之后的其余任何词都原样交给 git 处理。

## 跨 runtime

agit 把 Claude Code 与 Codex 当作两种 runtime。读取 session 的命令通过 `--from` 指定其一；只装了一个就用那个，两个都在而没有指定时便会询问。`agit adapter` 会列出已识别的 runtime。

`agit convert <src> --to <runtime>` 把一份 session 转换成另一 runtime 能够 resume 的格式，同 runtime 之间是字节级复制，跨 runtime 则是内容级转换，会舍弃没有对应物的部分。`agit resume <src>` 把一份 session 载入 runtime 并继续。守护进程的自动 convert 就是把这件事持续替你做着。

## 合并

`agit a merge <target>` 并不是按文本 diff。它把双方各自最新的 session 都以只读方式复活，让它们对照真实代码审视各自做过什么，尽可能完成对齐，最终产出一个**可以 resume 的合并后 session**，用 `claude --resume <id>` 或 `codex exec resume <id>` 接着做，同时把真正冲突之处列出来交给使用者裁决。它的产物是一段 session，而不是写进某个文件的摘要。

合并模式取决于身份。如果目标的 aid 与本地相同，例如同事推送上来的同一个 agent，就先经对话对齐，再把两侧的 git 历史也 merge 到一起；如果 aid 不同，也就是另一个 agent，就只做对话，两份记忆都完整保留。由于这个过程借助模型完成，它本身是非确定性的，这是换取一种无需维护 schema 的真正语义合并所付出的代价；而原始 session 始终是 git 里那份确定的事实来源。

## 协作与 hub

分享通过原生 git 完成，为 store 配置一个 remote 再 push 即可：

```bash
agit a remote add origin https://hub.example.com/frontend.git
agit a push -u origin HEAD          # 把 remote 记入 .agit.toml，凭据被剥除
git add .agit.toml && git commit    # 提交这个绑定，同事才能获得这个 agent
```

同事在已经克隆好的代码仓库里，一条 `agit a clone frontend` 就能依据 `.agit.toml` 把对应 agent 取回来；或者直接执行 `agit init`，它会把绑定中声明的每个 agent 逐一克隆到位。如果这个 agent 还带着自己的 harness，也就是它捕获的 MCP server、skills 与配置，`agit harness apply` 可以把它们一并落到本地。

如果要给团队一个带 web 界面和权限管理的服务端，可以自行部署 **agit-hub**。它把 agent 作为裸 git 仓库托管，提供同步，并内置一个编译进二进制的 React web UI，把每个 session 呈现为一条可浏览的事件时间线，附带来源信息与 revision diff。权限以 agent 为单位划分，包括 owner、可见性（默认私有），以及具备 read / write / admin 的成员；用户以密码登录，脚本和 git 则使用可以限定到单个 agent、可以设置有效期、也可以撤销的 token；每一个请求，包括 git smart-http，都经过同一处授权判定。每次 push 都会在服务端扫描密钥，因此泄漏的凭据无法进入共享仓库。agit-hub 单独分发，可通过 Docker 或源码构建，它只负责托管与渲染，既不运行 agent，也不执行合并，合并始终在客户端本地完成。

## git shadow

`agit shadow install` 会在你的 shell 里把 `git` 接到 `agit` 上，支持 bash、zsh、fish 与 PowerShell，于是每一条 git 命令都会顺带为 agent 上下文做版本控制。想恢复原样，执行 `agit shadow uninstall` 即可。

## 安全

session 里可能夹带密钥，比如 agent 读取过的 `.env` 或打印出来的 token，因此每个 store 在 commit 与 push 之前都会被扫描，推送到 hub 之后还会再扫描一次。

`.agit.toml` 则是他人撰写、攻击者可控的输入，所以 agit 在克隆它所声明的 remote 之前，会先把它和一份传输协议白名单比对，因为 `git clone 'ext::<cmd>'` 会执行其中的 `<cmd>`。

## 安装

```bash
npm install -g @einsia/agentgit
```

装好的是 `agit` 客户端。agit-hub 服务端单独通过 Docker 或源码构建获取。安装完成后可以选择执行 `agit shadow install`。
